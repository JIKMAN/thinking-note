# Q. Binlog 기반 CDC를 쓸 때 고려해야 할 것들은?

binlog CDC는 DB의 변경 이벤트(INSERT/UPDATE/DELETE)를 실시간으로 캡처하는 방식임.
Debezium 같은 툴이 MySQL binlog를 읽어서 Kafka로 흘리는 구조가 일반적임.

```
MySQL binlog
    ↓
Debezium (Kafka Connect)
    ↓
Kafka Topic (per table)
    ↓
Consumer (Spark Streaming / Flink / etc.)
    ↓
Data Lake (Iceberg)
```

---

## Q. Binlog CDC와 JDBC 증분 배치의 차이는 무엇인가요?

| 항목 | JDBC + updated_at | Binlog CDC |
|---|---|---|
| 지연 | 배치 주기 (수분~수시간) | 초 단위 |
| Hard Delete 감지 | 불가 | 가능 |
| 원천 DB 부하 | 배치 실행 시 쿼리 부하 | 거의 없음 (binlog 읽기) |
| 구현 복잡도 | 낮음 | 높음 |
| 인프라 의존성 | JDBC만 있으면 됨 | Kafka, Debezium 필요 |
| 원천 요건 | updated_at 컬럼 필요 | binlog_format=ROW 필요 |
| Exactly-once | upsert로 보완 가능 | 별도 처리 필요 |

---

## Q. Debezium이 Kafka에 이벤트를 쌓을 때 포맷이 어떻게 되나요?

Debezium은 변경 이벤트를 JSON 또는 Avro로 발행함. 각 이벤트에는 `before` / `after` 필드가 있음.

```json
{
  "op": "u",        // c=INSERT, u=UPDATE, d=DELETE, r=READ(snapshot)
  "before": { "id": 1, "status": "PENDING" },
  "after":  { "id": 1, "status": "CLOSED" },
  "source": {
    "db": "orders_db",
    "table": "orders",
    "ts_ms": 1711900000000
  }
}
```

- `op=d`(DELETE)일 때는 `after`가 null이고 `before`에 삭제된 행이 담김
- 이 구조 덕분에 Hard Delete도 감지 가능함

---

## Q. Binlog CDC에서 Exactly-once를 어떻게 보장하나요?

Kafka + Debezium 구조는 기본적으로 at-least-once임. Consumer가 실패하면 같은 이벤트를 다시 소비할 수 있음.

**보완 방법**
- **Iceberg MERGE INTO (upsert)**: PK 기준으로 이미 있는 행은 UPDATE, 없으면 INSERT. 이벤트가 중복 소비돼도 결과가 동일하므로 idempotent함.
- **Kafka offset 커밋 시점 관리**: 적재 성공 후에 offset을 커밋함. 적재 실패 시 offset을 커밋하지 않으면 해당 이벤트를 재처리함.
- **Kafka Transactions**: Producer 레벨에서 트랜잭션을 걸면 Kafka 내에서 exactly-once를 보장하지만, Consumer→Iceberg 구간은 별도로 처리해야 함.

완전한 Exactly-once는 어렵고, 실무에서는 idempotent upsert + offset 관리 조합으로 effectively-once를 구현하는 게 일반적임.

## Q. Binlog가 중간에 유실되거나 연결이 끊기면 어떻게 되나요?

Debezium은 binlog position(또는 GTID)을 Kafka Connect의 offset 저장소에 기록함. 재시작 시 마지막으로 처리한 position부터 이어서 읽음.

**문제가 생기는 케이스**
- **Binlog 보존 기간 초과**: MySQL은 기본 binlog 보존 기간이 짧음(기본 30일, 설정에 따라 더 짧음). Debezium이 오랫동안 멈췄다가 재시작하면, 그 사이의 binlog가 이미 삭제돼 있을 수 있음.
- **이 경우 Debezium은 Snapshot 모드로 전환**: 현재 원천 테이블을 전체 스냅샷으로 다시 읽은 후 그 시점부터 binlog를 이어받음. 이 과정에서 원천 DB에 일시적 부하가 생김.

**보완 방법**
- MySQL `binlog_expire_logs_seconds`(또는 `expire_logs_days`) 값을 충분히 길게 설정하도록 원천 팀에 요청함
- Debezium 연결 상태를 모니터링해서 오랫동안 끊기지 않도록 알림 설정함
- 재스냅샷 발생 시 Iceberg에 중복 데이터가 들어올 수 있으므로 upsert 구조 필수

## Q. 원천 DB 스키마가 바뀌면 Debezium은 어떻게 되나요?

스키마 변경이 binlog에 DDL 이벤트로 기록됨. Debezium은 이를 감지하고 내부 스키마 히스토리를 갱신함.

**케이스별 동작**
- **컬럼 추가**: Debezium이 자동으로 감지하고 새 컬럼을 이벤트에 포함시킴. 다운스트림(Iceberg)에서 mergeSchema=true면 자동 반영됨.
- **컬럼 삭제**: 이후 이벤트에서 해당 컬럼이 빠짐. Iceberg에는 기존 데이터가 그대로 남아 NULL로 읽힘.
- **타입 변경**: Debezium이 감지는 하지만 Consumer 쪽에서 역직렬화 실패가 날 수 있음. 특히 Avro + Schema Registry 조합이면 호환성 체크에서 걸림.

**보완 방법**
- Kafka Schema Registry의 호환성 모드를 `BACKWARD`로 설정하면 컬럼 추가만 허용되고 타입 변경/컬럼 삭제는 등록 거부됨
- 타입 변경이 감지되면 Slack 알림 + Consumer 중단 후 수동 확인하는 구조를 씀

## Q. 원천 DB에서 Bulk Update나 대량 INSERT가 발생하면 어떻게 되나요?

binlog에 행 단위로 이벤트가 기록됨. 100만 건을 한 번에 UPDATE하면 100만 개의 이벤트가 Kafka에 쌓임.

**문제**
- Kafka 토픽이 순간적으로 폭증함
- Consumer 처리량이 따라가지 못하면 Lag이 쌓임
- Iceberg에 소량씩 커밋이 반복되면 소파일(small file) 문제 발생

**보완 방법**
- Consumer에서 마이크로배치로 묶어서(예: 10초 또는 N천 건 단위) Iceberg에 커밋함. 건별 커밋보다 훨씬 효율적임.
- Kafka Consumer Lag 모니터링을 걸어두고 임계치 초과 시 알림
- Iceberg compaction(OPTIMIZE) 주기적으로 실행해서 소파일 정리

---

## Q. Kafka Consumer Lag이 쌓이면 어떻게 대응하나요?

Lag이 쌓이는 건 Consumer 처리량 < Producer 생산량인 상황임.

**원인 파악부터**
- 원천에서 대량 변경이 터진 건지 (일시적)
- Consumer 처리 로직이 느린 건지 (구조적)
- Iceberg 커밋 비용이 큰 건지

**대응**
- 일시적: 그냥 기다림. Lag이 빠지는지 추이를 봄.
- 구조적: Consumer 파티션 수 늘리거나 배치 사이즈 조정
- Iceberg 커밋 비용: 마이크로배치 주기 늘리거나 compaction 주기 조정
- SLA가 있는 테이블이면 Lag 임계치 알림 + 자동 스케일아웃

## Q. GTID vs binlog position, 어떤 걸 써야 하나요?

| 항목 | binlog position | GTID |
|---|---|---|
| 구성 | 파일명 + offset | 전역 고유 트랜잭션 ID |
| Failover | 복잡함 (Replica마다 position이 다름) | 간단함 (GTID는 서버 무관하게 동일) |
| 권장 | 단순 구성에서 무난 | HA 구성, Read Replica 사용 시 권장 |

Debezium은 두 방식 모두 지원함. HA나 Read Replica를 쓰는 환경이면 GTID가 훨씬 안전함. Failover 시 position 재계산 없이 그냥 이어받을 수 있음.

## Q. Snapshot 단계와 스트리밍 단계를 어떻게 구분하고 관리하나요?

Debezium 최초 실행 시 `snapshot.mode` 설정에 따라 동작이 달라짐.

| 모드 | 동작 |
|---|---|
| `initial` | 전체 스냅샷 후 binlog 스트리밍 시작 (기본값) |
| `never` | 스냅샷 없이 binlog부터 시작 (테이블이 이미 적재돼 있을 때) |
| `when_needed` | offset이 없거나 binlog가 유실된 경우에만 스냅샷 |
| `schema_only` | 스키마만 스냅샷, 데이터는 이후 변경분부터 |

**실무에서**
- 최초 적재는 `initial`로 전체 스냅샷 뜬 후 스트리밍으로 전환
- 이후 재시작은 `when_needed`로 설정해서 불필요한 스냅샷을 방지
- 스냅샷 구간에서 들어온 이벤트와 초기 적재 데이터가 중복될 수 있으므로 upsert 구조 필수

## Q. Binlog CDC의 한계는 무엇인가요?

솔직하게 말할 수 있어야 함.

- **원천 DB 요건**: `binlog_format=ROW` 설정이 필요함. DBA나 원천 팀 협조 없이는 못 씀.
- **인프라 복잡도**: Kafka, Kafka Connect, Debezium 운영이 추가됨. JDBC보다 관리 포인트가 많음.
- **이벤트 순서 보장**: 같은 PK에 대해 여러 이벤트가 빠르게 오면, Consumer가 병렬 처리할 때 순서가 바뀔 수 있음. Kafka 파티션 키를 PK로 설정해서 같은 PK는 같은 파티션으로 보내야 함.
- **트랜잭션 원자성**: 하나의 트랜잭션이 여러 테이블에 걸쳐 있으면, Kafka에서는 테이블별 토픽으로 분리됨. 두 토픽 간 원자적 소비가 어려움.
- **운영 난이도**: Debezium 버전 업, Kafka 운영, 스키마 변경 대응 등 JDBC 대비 운영 부담이 큼.

JDBC 증분 배치가 단순하고 충분한 유스케이스라면 굳이 CDC를 도입할 이유는 없음. 실시간성이 필요하거나 Hard Delete를 잡아야 하는 경우에 선택하는 방식임.
