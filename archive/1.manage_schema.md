# Q. 데이터 레이크 계층에서 스키마 관리를 어떻게 하는가?

Iceberg 기반으로 Schemaless로 적재하되 변경을 감지하고 통제하는 구조로 설계함.
엄격한 통제는 신뢰도가 높지만 유연함이 부족하기 때문에, 레이어별로 허용 수준을 다르게 가져가는 방식을 씀.

---

## 레이어 구조

```
원천
 ↓
Raw Zone     → Schemaless, 원천 원형 보존, mergeSchema=true
 ↓
Staged Zone  → 컬럼 추가만 허용, 타입 변경은 차단
 ↓
Curated Zone → 타입 강제, NOT NULL, 비즈니스 규칙 적용
```

---

## 스키마 변경 감지 및 대응 (Airflow DAG 기준)

| 변경 유형 | 대응 |
|---|---|
| 컬럼 추가 | Slack 알림 + 적재 허용 (nullable이라 안전) |
| 컬럼 삭제 | Slack 알림 + 수동 확인 요청 |
| 타입 변경 | 파이프라인 중단 + DLQ 라우팅 |

필요시 `meta.schema_changelog`에 히스토리 기록함.

---

## 케이스별 흐름

**1. 평상시**
Gate 통과 → 그냥 적재 → 아무 알림 없음

**2. 컬럼 추가 감지**
Slack 알림: `"orders에 new_col(STRING) 추가됨"`
→ 자동 적재 허용
→ changelog INSERT
→ 담당자가 나중에 확인 후 다운스트림 반영 여부 결정

**3. 타입 변경 / 컬럼 삭제 감지**
Slack 알림: `"🚨 orders.amount 타입 변경 INT→STRING"`
→ DAG 실패 처리 (적재 안 함)
→ 담당자가 원천 확인 후 수동으로 승인하면 재실행

---

## Q. 왜 Strict 스키마를 DI에 적용하지 않는가?

### 원천은 내가 통제할 수 없음
원천 DB는 보통 다른 팀, 다른 서비스가 관리함. 그쪽에서 컬럼 하나 추가했을 때 내 파이프라인이 터지면, 원천 변경 → 내 파이프라인 수정 → 배포 사이클이 생김. 원천 팀 배포 속도에 내 파이프라인이 종속되는 구조가 됨.

### 컬럼 추가는 사실 위험하지 않음
Iceberg 기준으로 컬럼이 추가되면 기존 파티션에서는 NULL로 읽힘. 다운스트림이 `SELECT *`를 쓰지 않는 한, 기존 쿼리는 그냥 동작함. 이걸 파이프라인 중단으로 막는 건 과잉 대응임.

### Strict 스키마의 비용
- Schema Registry 운영 비용(오버스펙)
- 원천 변경마다 스키마 버전 올리고 호환성 체크
- 파이프라인 배포 사이클 증가
- 원천 팀과의 조율 비용

이 비용을 DI 레이어에서 지불하면서 얻는 건 사실상 "잘못된 데이터를 조금 더 빨리 인지하는 것" 뿐임.

### DI의 데이터는 재처리가 가능함
원천을 Raw에 보존해두기 때문에, 스키마 변경이 나중에 발견되더라도 해당 시점부터 재처리하면 됨. 원천이 사라지는 게 아니니까 돌이킬 수 없는 손실이 아님.

### 감지 시점이 충분히 빠름
인제스트 앞단 Gate Task에서 잡으면 배치 단위로 감지됨. 문제가 있는 데이터가 Curated까지 올라가기 전에 차단되는 구조면, 비즈니스 영향이 발생하기 전에 대응 가능함.

🔥 예상 압박 질문 1: "컬럼 삭제 시 '수동 확인 요청'만 한다고 하셨는데, 그 사이에 Curated나 뷰(View)를 말아주는 다운스트림 DAG가 돌면 거기는 다 터지는 거 아닌가요? 데이터 정합성이 깨질 텐데요."
면접관의 의도: "알림망만 쳐두고, 뒷단 파이프라인 터지는 건 방치하는 거 아님?"
💯 베스트 방어 답변:
"맞습니다. 삭제된 컬럼을 필수적으로 참조하는 다운스트림(Curated) 파이프라인이라면 당연히 실패(Fail)하게 됩니다. 하지만 여기서 핵심은 **'장애 지점(Fail Point)을 어디에 두느냐'**입니다. 만약 DI 단(Raw)에서 막아버리면, 삭제된 컬럼과 무관한 다른 분석 지표들마저도 데이터 적재가 멈춰버리는 1차 병목이 생깁니다. 저희 방식은 영향을 받지 않는 파이프라인은 정상적으로 흐르게 두고, **해당 컬럼을 참조하는 타겟 DAG만 우아하게 실패(Graceful Failure)**하도록 두는 것입니다. 실패 알림을 받고 분석가와 협의하여 제외된 컬럼 로직을 Curated 쿼리에서 제거하면 시스템 전체의 멈춤 현상(Single Point of Failure)을 막을 수 있습니다."

🔥 예상 압박 질문 2: "타입 변경 시 파이프라인을 중단하고 DLQ(Dead Letter Queue)로 보낸다고 했는데, 그럼 수동으로 스키마 타입 맞춰서 재처리할 때까지 데이터가 엄청 밀릴 텐데 실시간성에 지장 없나요?"
면접관의 의도: "스키마 꼬였다고 파이프라인 껐다 켜면 파이프라인 지연(SLA 위반)은 어떻게 감당할 건가요?"
💯 베스트 방어 답변:
"타입 변환(예: INT에서 STRING으로 강제 변경)과 같은 치명적인 스키마 변경은 데이터 파이프라인 생태계에서 **'타협 불가능한 장애'**로 취급하는 것이 맞다고 생각합니다. 데이터가 밀리는 지연(SLA 위반)보다 더 치명적인 것은, 오염된 쓰레기 데이터(Garbage)가 Curated까지 흘러 들어가 경영진이 보는 비즈니스 지표가 틀어지는 것입니다. 데이터 밀림 현상은 로깅된 DLQ 파티션을 멱등하게 Backfill(재처리)하여 빠르게 해소할 수 있도록 클러스터 자원(EMR Auto-scaling)과 Airflow 재실행 구조를 최적화하는 데 리소스를 투자하여 극복하고 있습니다."

🔥 예상 압박 질문 3: "Iceberg의 Schema Evolution 기능만 믿고 파이프라인을 구축하셨는데, 중앙 집중형 Schema Registry (Confluent / Protobuf / Avro 등) 시스템을 쓰지 않은 이유는 무엇인가요? 돈 아끼려고 그런 건가요?"
면접관의 의도: "엔터프라이즈급이면 Schema Registry 쓰는 게 정석인데 인프라 설계 능력이 부족한 거 아닐까?"
💯 베스트 방어 답변:
"조직의 현재 성숙도와 백엔드 원천팀과의 결합도를 고려한 실용적 선택(Trade-off)이었습니다. Schema Registry가 제 기능을 하려면, 백엔드 개발자가 원천 DB 컬럼을 추가/수정하여 배포할 때 CI/CD에서 Registry의 호환성 검사를 통과해야만 배포되게끔 전사적인 강결합(Data Contract) 거버넌스가 도입되어야 합니다. 데이터팀이 이걸 강제할 수 없는 조직 구조에서는, Registry를 세팅해 봐야 '감지' 용도의 비싼 알림망에 불과합니다. 따라서 현재는 Iceberg 기반의 Lightweight한 변화 감지 + meta.schema_changelog를 통한 이력 관리로 비용 대비 최고의 효율을 내고 있습니다. 향후 조직이 더 커져서 명시적인 Data Contract 체계가 도입된다면 그때는 당연히 Registry 아키텍처로 진화할 것입니다."

🔥 예상 압박 질문 4: "컬럼 추가(Add) 시 자동 허용하면, 기존 파티션(과거 데이터)에서는 해당 컬럼이 NULL로 나올 텐데, 다운스트림 데이터를 사용하는 분석가(DA)나 데이터 사이언티스트(DS)들이 혼동하지 않나요?"
면접관의 의도: "데이터 엔지니어(DE)가 파이프라인 살리려고 다운스트림 유저(DA/DS)들에게 책임을 떠넘기는 거 아닌가요?"
💯 베스트 방어 답변:
"실제로 분석 시점의 혼선이 발생할 수 있습니다. 이를 방지하기 위해 기술적 허용과 별개로 **Data Discoverability(데이터 가시성)**를 함께 제공합니다. Airflow 감지 파이프라인에서 단순히 통과만 시키는 게 아니라, Slack 데이터 관련 채널에 [Notice] A 테이블에 X 컬럼이 오늘 날짜(YYYY-MM-DD) 파티션부터 추가되었습니다. 라고 방송을 해주고 있습니다. 또한 meta.schema_changelog 이력을 카탈로그 단에 노출하여, 분석가가 해당 컬럼을 SELECT 할 때 '아, 이 값은 5월 1일 전에는 원래 Null인 게 맞구나' 라고 스스로 인지할 수 있는 컨텍스트 환경(Context-awareness)을 만들어 혼란을 최소화하고 있습니다."
