# Spark 핵심 개념

---

## MapReduce vs Spark 실행 구조

| | MapReduce | Spark |
|---|---|---|
| 실행 단위 | Mapper 1개 = JVM 프로세스 1개 | Executor 1개(JVM) = Task N개 (멀티스레드) |
| JVM 비용 | Task마다 JVM 기동/종료 | JVM 재사용 |
| 중간 결과 | 매 단계마다 HDFS에 저장 | 메모리 유지 (셔플 경계에서만 디스크) |

---

## 인메모리 처리

Spark가 빠른 이유는 중간 결과를 메모리에 유지하기 때문임.

```
MapReduce
  Map → HDFS write → Reduce → HDFS write → Map → ...
  (단계마다 디스크 I/O)

Spark
  Stage 1 → 메모리 → Stage 2 → 메모리 → Stage 3
  (셔플 경계에서만 디스크)
```

단, 조건이 있음:
- 데이터가 메모리에 다 들어가야 함. 넘치면 디스크로 spill
- 셔플은 Spark도 디스크 + 네트워크 I/O 발생
- 단순 1회성 ETL은 인메모리 이점이 크지 않음
- `cache()` / `persist()`로 명시해야 메모리에 올려둠

---

## Narrow vs Wide Transformation

**Narrow**: 파티션 하나 → 파티션 하나. 다른 파티션 데이터 불필요. 셔플 없음.

```python
filter(), select(), map(), flatMap(), union()
```

**Wide**: 여러 파티션 데이터가 합쳐짐. 셔플 발생. Stage 경계가 됨.

```python
groupBy(), join(), orderBy(), distinct(), repartition()
```

```
read → filter → map → groupBy → filter → orderBy → write
       [  Stage 1  ]   ↕셔플   [ Stage 2 ]  ↕셔플  [Stage 3]
```

Stage 내부(Narrow)는 중간 저장 없이 파이프라인으로 처리.
Stage 경계(Wide)에서만 디스크 write → 네트워크 전송 → 디스크 read.

---

## 셔플 비용

셔플은 Spark에서 가장 비싼 연산임.

```
Shuffle Write (보내는 쪽)
  메모리 처리 결과 → 키 기준 정렬 → 로컬 디스크 write

네트워크 전송
  각 노드가 필요한 파티션을 HTTP로 가져감

Shuffle Read (받는 쪽)
  네트워크로 받은 데이터 → 로컬 디스크 저장 → 읽어서 다음 Stage 실행
```

→ 디스크 I/O + 네트워크 I/O 모두 발생. 셔플 최소화가 Spark 튜닝의 핵심.

## Lazy Evaluation

Transformation은 호출 즉시 실행되지 않음. **Action이 호출되는 순간 DAG 전체가 실행됨**.

```python
df2 = df.filter("status = 'ACTIVE'")   # 실행 안 됨
df3 = df2.groupBy("user_id").count()   # 실행 안 됨
df3.show()  # 여기서 처음으로 전부 실행
```

**주요 Action**

| Action | 설명 |
|---|---|
| `show()` | 결과 일부 출력 |
| `count()` | 행 수 반환 |
| `collect()` | 전체 결과를 드라이버로 가져옴 |
| `write` | 결과 저장 |
| `take(n)` | 상위 n개 |

---

## Lazy Evaluation의 장점

**1. 전체 DAG를 보고 최적화**
```python
df.filter("dt = '2024-01-01'")
  .filter("status = 'ACTIVE'")
  .select("id", "amount")
  .groupBy("id").sum("amount")
  .show()
```
- filter 두 개 → 하나로 합침 (predicate pushdown)
- select 있으니 두 컬럼만 읽음 (column pruning)
- Parquet이면 해당 컬럼 블록만 읽음 → I/O 자체 감소

**2. 불필요한 연산 제거**
```python
df2 = df.groupBy("user_id").count()
if some_condition:
    df2.show()  # False면 groupBy 셔플 자체가 발생 안 함
```

**3. Narrow transformation은 파이프라인으로 처리**

중간 결과 저장 없이 레코드 하나가 Stage 내 전체 연산을 통과함. 메모리와 I/O 모두 절약.

## 실행 계획 분석

Spark는 SQL/DataFrame 작업을 실행하기 전에 4단계 실행 계획을 만듦.

```python
df.filter("status = 'ACTIVE'").groupBy("user_id").count().explain(True)
```

```
== Parsed Logical Plan ==      ← 코드를 그대로 파싱한 것. 최적화 없음
== Analyzed Logical Plan ==    ← 컬럼/테이블 이름 검증, 타입 확인
== Optimized Logical Plan ==   ← Catalyst 옵티마이저가 최적화 (predicate pushdown 등)
== Physical Plan ==             ← 실제 실행 방식 결정 (어떤 Join 전략, 셔플 방식 등)
```

**실무에서는 Physical Plan만 봐도 충분한 경우가 많음**

```python
df.explain()  # Physical Plan만 출력
```

---

### 실행 계획에서 봐야 할 핵심 포인트

**1. Scan 단계 - 얼마나 읽나**

```
FileScan parquet db.orders
  PartitionFilters: [isnotnull(dt), (dt = 2024-01-01)]   ← 파티션 pruning 됐는지
  PushedFilters: [IsNotNull(status), EqualTo(status, ACTIVE)]  ← 필터 pushdown 됐는지
  ReadSchema: struct<id:bigint, amount:decimal>           ← 필요한 컬럼만 읽는지
```

- `PartitionFilters`가 있으면 파티션 pruning 동작 중 → 읽는 파일 수 감소
- `PushedFilters`가 있으면 Parquet 레벨에서 필터링 → 읽는 행 수 감소
- `ReadSchema`에 불필요한 컬럼이 없으면 column pruning 동작 중

**2. Join 전략 - 어떤 방식으로 조인하나**

| Physical Plan에 표시 | 의미 | 셔플 여부 |
|---|---|---|
| `BroadcastHashJoin` | 작은 테이블을 전 노드에 브로드캐스트 | 없음 (가장 빠름) |
| `SortMergeJoin` | 양쪽 다 정렬 후 병합 | 양쪽 모두 셔플 |
| `ShuffledHashJoin` | 한쪽 셔플 후 해시 조인 | 한쪽 셔플 |

```
# BroadcastHashJoin이 나오면 이상적
BroadcastHashJoin [user_id], [user_id], Inner, BuildRight
```

```
# SortMergeJoin이 나오면 양쪽 셔플 발생 → 큰 테이블 간 조인
SortMergeJoin [user_id], [user_id], Inner
```

**3. Exchange - 셔플 발생 위치**

```
Exchange hashpartitioning(user_id, 200)   ← 셔플 발생. 파티션 수 200
```

- `Exchange`가 보이면 셔플 발생 지점
- 파티션 수(200)가 너무 크면 소파일 문제, 너무 작으면 병렬성 저하

**4. Sort - 불필요한 정렬 없는지**

```
Sort [amount DESC], true   ← 정렬 발생. 셔플 후 각 파티션 내 정렬
```

- 최종 결과에 `orderBy`가 없는데 Sort가 보이면 불필요한 정렬
- `SortMergeJoin` 전에 Sort가 붙는 건 정상

### Spark UI로 실행 계획 분석

`explain()`은 텍스트 기반이지만, Spark UI는 시각적으로 병목을 보여줌.

```
http://driver-host:4040
```

**Jobs 탭**
- Action 단위로 Job이 생성됨
- 오래 걸리는 Job 식별

**Stages 탭**
- 셔플 경계마다 Stage가 나뉨
- `Shuffle Read/Write Size` 확인 → 너무 크면 데이터 skew 또는 불필요한 셔플
- `Tasks` 중 특정 Task만 오래 걸리면 → Data Skew 의심

**SQL 탭**
- DAG를 시각적으로 보여줌
- 각 노드에 처리한 행 수, 시간이 표시됨
- `PhotonScan` / `FileScan`에서 읽은 행 수 vs 필터 후 행 수 비교 → pruning 효과 확인

## 성능 튜닝 전략

### 1. Data Skew 해결

가장 흔하고 임팩트 큰 문제. 특정 파티션만 비대해서 해당 Task만 오래 걸림.

**증상**: Spark UI Stages 탭에서 Task 대부분은 1초인데 특정 Task만 10분

```python
# 원인 파악: 키별 데이터 분포 확인
df.groupBy("user_id").count().orderBy("count", ascending=False).show(20)
```

**해결 방법**

```python
# 방법 1: Salting - skew 키에 랜덤 prefix 붙여서 분산
from pyspark.sql.functions import concat, lit, rand, floor

df_salted = df.withColumn("salted_key",
    concat(col("user_id"), lit("_"), floor(rand() * 10).cast("string")))

# 방법 2: skewHint (Spark 3.x AQE)
df.hint("skew", "user_id").join(other, "user_id")

# 방법 3: AQE 활성화 (Spark 3.x)
spark.conf.set("spark.sql.adaptive.enabled", "true")
spark.conf.set("spark.sql.adaptive.skewJoin.enabled", "true")
# → Spark가 런타임에 skew 파티션을 자동 분할
```

### 2. 셔플 최소화

```python
# 나쁜 예: 같은 키로 두 번 groupBy → 셔플 2번
df.groupBy("user_id").agg(sum("amount"))
df.groupBy("user_id").agg(count("order_id"))

# 좋은 예: 한 번에 처리 → 셔플 1번
df.groupBy("user_id").agg(
    sum("amount").alias("total_amount"),
    count("order_id").alias("order_count")
)
```

```python
# 나쁜 예: 큰 테이블 join → SortMergeJoin (셔플 2번)
big_df.join(big_df2, "user_id")

# 좋은 예: 작은 테이블이면 broadcast → 셔플 없음
from pyspark.sql.functions import broadcast
big_df.join(broadcast(small_df), "user_id")
# 기준: spark.sql.autoBroadcastJoinThreshold (기본 10MB)
```

---

### 3. 파티션 수 조정

```python
# 셔플 후 파티션 수 (기본 200, 데이터 크기에 맞게 조정)
spark.conf.set("spark.sql.shuffle.partitions", "400")

# 적정 파티션 수 기준: 파티션 하나당 100~200MB
# 총 데이터 크기 / 128MB = 적정 파티션 수

# 파티션이 너무 많으면: 소파일 + 태스크 오버헤드
df.coalesce(10)      # 줄이기 (셔플 없음, 불균형 가능)
df.repartition(50)   # 늘리거나 균등 분산 (셔플 발생)
```

### 4. Predicate Pushdown / Column Pruning 확인

Spark가 자동으로 해주지만, 안 되는 경우가 있음.

```python
# 나쁜 예: UDF를 filter에 쓰면 pushdown 안 됨
from pyspark.sql.functions import udf
is_active = udf(lambda x: x == 'ACTIVE')
df.filter(is_active(col("status")))  # pushdown 불가

# 좋은 예: 내장 함수 사용
df.filter(col("status") == 'ACTIVE')  # pushdown 가능
```

```python
# 파티션 컬럼으로 filter해야 파티션 pruning 동작
df.filter("dt = '2024-01-01'")   # 파티션 pruning O
df.filter("amount > 1000")        # 파티션 pruning X (파티션 컬럼 아님)
```

### 5. Cache 전략

```python
# 같은 DataFrame을 여러 번 쓸 때만 cache
df_base = df.filter("status = 'ACTIVE'").cache()

result1 = df_base.groupBy("user_id").count()
result2 = df_base.groupBy("dt").sum("amount")
# df_base가 cache 없으면 두 번 읽음. cache 있으면 한 번만 읽음

# 다 쓰고 나면 반드시 해제
df_base.unpersist()
```

```python
# 저장 레벨 선택
from pyspark import StorageLevel

df.persist(StorageLevel.MEMORY_ONLY)          # 메모리만 (빠름, OOM 위험)
df.persist(StorageLevel.MEMORY_AND_DISK)      # 메모리 부족 시 디스크로 (안전)
df.persist(StorageLevel.DISK_ONLY)            # 디스크만 (느림, 메모리 부족 시)
```

### 6. AQE (Adaptive Query Execution, Spark 3.x)

런타임 통계를 보고 실행 계획을 동적으로 변경함. Spark 3.x에서 기본 활성화.

```python
spark.conf.set("spark.sql.adaptive.enabled", "true")
```

AQE가 해주는 것:
- **동적 파티션 병합**: 셔플 후 소파일 파티션을 자동으로 합침
- **동적 Join 전략 변경**: SortMergeJoin으로 계획했다가 런타임에 데이터가 작으면 BroadcastHashJoin으로 전환
- **Skew Join 자동 처리**: skew 파티션을 자동으로 분할

---

### 튜닝 우선순위 요약

```
1. explain()으로 실행 계획 확인 → 불필요한 셔플, Full Scan 없는지
2. Spark UI로 Data Skew 확인 → Task 시간 편차가 크면 skew
3. 셔플 최소화 → Broadcast Join, 중복 groupBy 제거
4. 파티션 수 조정 → shuffle.partitions, repartition/coalesce
5. AQE 활성화 → Spark 3.x면 기본 켜두기
6. Cache → 반복 사용 DataFrame에만 선택적으로
```
