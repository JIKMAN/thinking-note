# Q. 데이터 파이프라인(EMR 기반)에서 테스트 코드는 어떻게 활용하고 있나요? (특히 단위 테스트)

## A. 핵심 요약
"현재 운영 환경에서 EMR을 활용해 데이터 레이크의 원천 데이터를 2차 가공하는 작업을 주로 하고 있습니다. 결론부터 말씀드리면, **전통적인 방식의 단위 테스트(Unit Test)는 적극적으로 작성하지 않고 있습니다.** 
물론 단위 테스트가 런타임 오류나 코드 레벨의 예외 케이스를 잡는 데 도움이 될 수 있다는 점은 인지하지만, 데이터 파이프라인의 특성과 비용 대비 효용성을 고려했을 때 **단위 테스트보다는 데이터 품질 테스트(Data Quality Test)와 통합/E2E 테스트에 더 집중**하고 있습니다."

---

## 단위 테스트를 사용하지 않는 합리적인 이유 (설득 논리)

### 1. 장애의 주원인은 '로직'보다 '데이터 상태와 분산 환경'
웹/앱 애플리케이션 개발에서는 분기문이나 복잡한 비즈니스 로직의 결함이 버그를 만듭니다. 하지만 데이터 엔지니어링(Spark/EMR)에서의 2차 가공 로직은 `SELECT`, `JOIN`, `GROUP BY` 등 선언적(Declarative)인 경우가 많습니다.
실제 파이프라인 배치가 터지는 주된 이유는 코드 로직의 결함보다는 **예상치 못한 원천 데이터의 형태(Null, 스키마 변경, 비정상적인 값)**나 **분산 환경의 한계(데이터 쏠림으로 인한 OOM 등)**에서 비롯됩니다. 소규모의 정제된 로컬 Mock 데이터로 통과하는 단위 테스트는 실제 운영의 10TB 데이터 무결성이나 OOM을 보장해주지 못합니다.

### 2. Mocking 비용 대비 낮은 효용성 (ROI)
단위 테스트를 작성하려면 Spark Session을 띄우고, 원천 데이터의 스키마와 특성을 재현하는 Mock 데이터를 만들어야 합니다. EMR 환경에 종속적인 플랫폼 설정이나 수십 개의 컬럼을 가진 Parquet/Iceberg 테이블 구조를 로컬에서 완벽히 Mocking하는 것은 작성 및 유지보수 비용(시간)이 매우 큽니다. 
이렇게 고비용을 들여 테스트를 만들어도, 정작 잡아내는 오류는 문법 오류나 단순 컬럼명 오타인 경우가 많아(이는 실제 실행을 통해 즉각 인지 가능) 투자 대비 효율(ROI)이 낮다고 판단했습니다.

### 3. 순수 로직과 인프라 의존성 로직의 분리 (테스트 철학)
파이프라인 테스트에 대한 저의 핵심 철학은 **"검증의 대상이 순수 로직인가, 인프라/데이터 의존성인가"**를 구별하는 것입니다.
만약 복잡한 커스텀 UDF(User Defined Function)나 설정 파일을 동적으로 파싱하는 유틸리티처럼 **외부 의존성이 덜한 순수 함수**가 있다면, 로컬에서의 빠른 피드백(Fast Feedback Loop)과 엣지 케이스 점검, 회귀(Regression) 방지를 위해 단위 테스트를 작성하는 것이 매우 유효하고 당연히 필요합니다. 
하지만 현재 제가 주로 다루는 2차 가공 로직들은 Spark의 선언적 API(`withColumn`, `join` 등)와 분산 환경(EMR/S3)에 극도로 밀착되어 있습니다. 이러한 환경 종속적인 코드를 억지로 로컬에서 Mocking 하여 단위 테스트를 작성하는 것은 사실상 '스파크 프레임워크 자체를 재테스트'하는 중복 작업에 불과합니다. **순수 로직은 단위 테스트로, 인프라/데이터 종속성이 높은 파이프라인의 핵심은 E2E 및 Data Quality 테스트로 검증한다**는 것이 제 확실한 기준입니다.

---

## 🚀 그럼 '테스트'와 '품질 보증'은 어떻게 하는가?
단위 테스트를 생략하는 대신, 데이터 엔지니어링 생태계에 더 적합한 관점으로 품질(Quality)과 안정성을 보장합니다.

### 1. 데이터 품질 테스트 (Data Quality Test / Data Contract)
코드 동작 자체를 검증하기보다, **가공이 끝난 데이터 결과물의 상태를 검증**합니다.
*   **Null & Schema Check**: PK 역할을 하거나 조인 키로 사용되는 핵심 컬럼에 Null이 예상 범위를 초과하지 않는지 확인합니다.
*   **Volume Anomaly**: 2차 가공 후 데이터 건수(Row count)가 전일 대비 혹은 평균 대비 비정상적으로 급증/급감하지 않았는지 체크합니다.
*   **Uniqueness**: 고유해야 하는 단위 row에 중복이 발생하지 않는지 검증합니다.
파이프라인의 특정 단계나 최종 적재 전에 이러한 제약(Assertion)을 걸어 오류 데이터가 다운스트림(Downstream)으로 전파되는 것을 차단합니다.

### 2. 운영 데이터 기반의 Staging 검증과 WAP 패턴 (Write-Audit-Publish)
새로운 로직이나 변경 사항을 검증할 때는 로컬에서의 단위 테스트보다, **운영과 동일한 데이터 카탈로그(Glue Catalog) 내의 Staging(또는 Dev) 영역에 데이터를 직접 적재하여 운영 파티션(혹은 샘플링 데이터)으로 E2E 테스트(Dry Run)**를 수행하는 방식을 선호합니다. 
이를 통해 실제 운영 스키마와의 호환성, 극단적인 데이터 분포, 클러스터 메모리 현황 등을 가장 정확하게 검증할 수 있으며, Staging 적재(Write) -> 품질 검증(Audit) -> 운영 배포(Publish)로 이어지는 데이터 무결성 패턴을 구축할 수 있습니다.
이러한 동일 카탈로그 내 검증 방식에서 발생할 수 있는 리스크들은 다음의 방향으로 관리하고 개선해 나갈 계획입니다.
*   **데이터 덮어쓰기 방지 (권한 분리)**: 현재는 테스트 런이 운영 데이터를 덮어쓰는 사고를 방지하기 위해 **저장되는 버킷을 다르게 하고 네이밍 컨벤션을 통해 물리적으로 분리**하고 있습니다. 향후에는 여기서 더 나아가 **IAM Role이나 AWS Lake Formation 등을 활용해, Staging Job이 운영 네임스페이스에 접근하거나 수정할 수 없도록 완벽한 권한 분리(Access Control) 체계를 구축**하는 것이 바람직하다고 생각합니다.
*   **스토리지 및 카탈로그 관리 (데이터 초기화)**: Staging 영역에 쌓이는 임시 테스트 데이터들은 **현재는 수동으로 정리(Drop/Delete)하며 관리**하고 있습니다. 향후에는 **S3 Lifecycle(수명 주기) 정책을 적극 활용하여, 일정 기한이 지난 테스트 데이터가 자동으로 소멸**되도록 구성함으로써 스토리지 비용 최적화와 카탈로그 관리의 자동화를 이룰 계획입니다.

### 3. 실패 허용 및 멱등성(Idempotency) 보장
파이프라인 자체를 실패에 강하고 유연하게 복구할 수 있도록 설계합니다. 로직 오류나 원천 데이터의 문제로 파이프라인이 중단되더라도 언제든 코드를 수정한 뒤 재실행(Backfill/Rerun)하면 중복 적재나 부작용 없이 동일하고 정확한 결과물을 낼 수 있도록 멱등성을 보장하는 데 엔지니어링 역량을 더 투자합니다.
