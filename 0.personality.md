# 인성 질문 답변

## 🔵 Loved By Customer (고객 중심)

**1. 본인이 만든 데이터 파이프라인이나 시스템이 실제 비즈니스(고객 또는 내부 팀)에 어떤 가치를 만들었는지 구체적인 사례를 말해주세요.**

분리되어 있던 소스들을 통합한 데이터레이크 기반의 분석 환경을 제공하여, 서비스DB나 운영 리소스에 종속되지 않은 별도의 분석 환경을 제공하고, 요구사항에 따라 데이터 마트나 대시보드 등 BI 지원을 통해 데이터 기반의 의사결정 문화 정착에 기여함.

**2. 기술적으로 올바른 방향과 고객(혹은 내부 사용자)이 원하는 방향이 충돌했을 때, 어떻게 판단하고 결정했나요?**

이런 충돌이 생기면 가장 먼저 "데이터 신뢰도를 지키는 방향이 어느 쪽인가"를 기준으로 판단함. 결국 데이터 팀의 존재 이유는 사용자에게 정확한 데이터를 제공하는 것이고, 그게 곧 데이터 팀에 대한 신뢰도로 이어지기 때문임.

스키마 변경 감지 시 파이프라인을 무조건 중단하는 Strict 방식을 처음에 제안했음. 기술적으로 안전하다고 생각했기 때문임. 그런데 실제로 적용해보니 분석팀에서 "어제 리포트가 왜 안 나와요?"라는 문의가 반복됐음. 원천 팀이 컬럼 하나 추가한 것 때문에 파이프라인이 멈춘 거였음.

"데이터가 안 들어오는 것"과 "잘못된 데이터가 들어오는 것" 중 어느 쪽이 신뢰도를 더 해치는지를 다시 생각했음. 컬럼 추가는 Iceberg 기준으로 기존 데이터에 영향을 주지 않기 때문에, 이걸 막는 건 데이터 정합성 문제가 아니라 운영 편의 문제였음. 반면 타입 변경은 실제로 데이터가 오염될 수 있는 케이스라 차단이 맞음.

컬럼 추가는 허용하되 타입 변경만 차단하는 방식으로 바꿨음. 사용자가 원하는 방향이라서 바꾼 게 아니라, 데이터 신뢰도 기준으로 판단했을 때 그게 맞는 방향이었기 때문임.

**3. 데이터 엔지니어로서 "고객 목소리"를 어떻게 업무에 반영하나요?**

직접 고객을 만나지 않더라도 내부 고객인 분석팀, 비즈니스팀이 어떤 데이터를 어떻게 쓰는지를 파악하는 게 중요하다고 생각함. 파이프라인 설계 전에 다운스트림 사용자가 어떤 쿼리 패턴을 쓰는지 먼저 확인함. 그래야 파티션 키, 적재 주기, 스키마 구조를 올바르게 결정할 수 있음. 데이터 엔지니어에게 고객 중심은 "내 파이프라인이 정상 동작하는가"가 아니라 "다운스트림이 원하는 데이터가 제때 제대로 도착하는가"라고 생각함.

## 🟠 Execution Over Perfection (실행 우선)

**4. 완벽하지 않은 상태로 먼저 배포하거나 실행했던 경험이 있나요?**

광고 도메인에 ClickHouse를 처음 도입했던 경험임. 초기에 운영 트래픽을 충분히 고려하지 못하고 메모리 리소스를 낮게 잡았음. 운영에 올리자마자 두 가지 OOM이 발생했음. 하나는 분석 쿼리 실행 시 메모리 초과, 다른 하나는 ClickHouse INSERT 시 메모리 초과였음.

쿼리 OOM은 ClickHouse의 `max_memory_usage` 설정이 실제 쿼리 패턴보다 낮게 잡혀있었고, INSERT OOM은 배치 사이즈를 너무 크게 잡아서 한 번에 올리는 데이터 양이 메모리를 초과한 게 원인이었음.

단기 대응으로는 INSERT 배치 사이즈를 줄이고 `max_memory_usage` 설정을 올렸음. 근본적으로는 실제 운영 쿼리 패턴과 INSERT 빈도를 모니터링한 뒤 리소스를 재산정해서 스펙을 조정했음.

완벽하게 준비되지 않은 채로 올라간 케이스였지만, 실제 트래픽 없이는 정확한 리소스 산정이 어렵기도 했음. 이후로는 신규 도입 시 초기 단계에 모니터링을 더 촘촘히 붙이고, 여유 메모리를 보수적으로 잡는 방식으로 접근하게 됐음.

**5. 기술 스택이나 아키텍처 선택 시, 최선과 실행 가능한 선택 사이 균형을 어떻게 잡나요?**

일단 현재 팀 규모, 운영 가능한 인프라, 요구되는 SLA를 기준으로 봄. Schema Registry + CDC가 기술적으로 더 완성도 높은 선택이지만, 팀 규모가 작고 운영 인력이 제한적이면 오히려 유지보수 부담이 더 큰 선택이 됨. "가장 좋은 기술"보다 "지금 우리가 운영 가능한 기술"을 먼저 고려함. 단, 나중에 스케일업이 필요한 지점을 미리 파악해두고 어느 시점에 어떤 기술로 전환할지 로드맵을 그려두는 방식으로 균형을 잡음.

**6. 불필요하다고 느낀 회의나 프로세스를 줄이거나 개선한 경험이 있나요?**

파이프라인 이슈가 생길 때마다 Slack 채널에 수동으로 상황을 공유하던 프로세스가 있었음. 사람이 직접 챙겨야 해서 야간 장애 시 누락되는 경우가 있었음. Airflow에 실패 시 자동 Slack 알림을 붙이고, 알림 메시지에 원인, 영향 범위, 재실행 방법을 포함시켰음. 수동 공유 프로세스 없이도 이해관계자가 즉시 파악할 수 있는 구조를 만들었음.

**7. 요구사항이 불명확한 상황에서 어떻게 일을 시작하나요?**

먼저 움직이는 편임. 불명확한 요구사항을 기다리다 보면 뒤로 밀리는 경우가 많기 때문임. 다만 무작정 시작하는 게 아니라, 내가 가정한 것들을 먼저 명시적으로 문서화하고 공유함. "이렇게 이해한 게 맞나요?"를 구두로 묻는 것보다, 가정을 적어서 보여주면 상대방이 틀린 부분만 정정해주는 방식이 더 빠르게 정렬이 됨. 잘못된 방향으로 너무 멀리 가기 전에 체크포인트를 짧게 가져가는 방식으로 진행함.

## 🔴 Bounce Back And Finish Strong (끝까지 해내기)

**8. 기술적으로 막혀서 장기간 해결이 안 됐던 문제가 있나요?**

Iceberg MERGE INTO 성능 문제가 있었음. 대용량 테이블에서 전체 파티션을 스캔하면서 처리 시간이 기하급수적으로 늘었음. 처음에는 Spark 리소스 튜닝으로 해결하려 했는데 근본적인 해결이 안 됐음. 실행 계획을 분석하다가 MERGE 조건절에 파티션 필터가 없어서 전체 스캔이 발생한다는 걸 파악했음. `ON` 절에 파티션 컬럼 필터를 명시적으로 추가하자 스캔 범위가 최근 파티션으로 한정됐고, 처리 시간이 대폭 줄었음. 도구를 믿기 전에 실행 계획을 먼저 보는 습관이 생긴 계기였음.

**9. 일정 압박 상황에서도 퀄리티를 포기하지 않았던 경험, 어떻게 "충분히 좋다"를 판단했나요?**

파이프라인 오픈 일정이 당겨졌을 때, 전체 레이어(Raw → Staged → Curated)를 한 번에 완성하기 어려운 상황이었음. Raw → Staged까지만 먼저 오픈하고 Curated는 이후에 추가하는 방식으로 조정했음. "충분히 좋다"의 기준은 다운스트림이 실제로 쓸 수 있는 데이터가 나오는가였음. 타입 검증이나 비즈니스 룰이 없는 Staged 데이터라도, 분석팀이 당장 쓸 수 있고 Known Limitation을 알고 있다면 오픈할 수 있다고 판단했음. 대신 Curated 완성 일정을 명시적으로 공유했음.

**10. "적당한 결과"와 "더 나은 결과"의 기준은 무엇인가요?**

적당한 결과는 요구사항을 충족하는 것, 더 나은 결과는 다음 사람이 유지보수하기 쉬운 것이라고 생각함. 파이프라인이 돌아가더라도 장애 시 원인 파악에 30분이 걸리면 충분하지 않다고 봄. 알림 메시지에 컨텍스트를 넣고, 재실행 방법을 문서화하고, 실행 계획을 설명하는 주석을 남기는 것들이 "더 나은 결과"를 만드는 작업이라고 생각함. 나 혼자 이해하는 파이프라인은 팀 자산이 아님.

**11. 프로젝트 중간에 예상치 못한 기술적 장애나 요구사항 변경이 생겼을 때 어떻게 대응했나요?**

원천 팀이 예고 없이 테이블 타입을 변경해서 파이프라인이 중단된 적 있었음. 당장의 대응은 해당 배치를 DLQ로 라우팅하고 알림을 보내는 것이었음. 이후에는 스키마 변경 감지 Gate를 파이프라인 앞단에 추가해서, 타입 변경이 발생하면 자동으로 중단하고 담당자에게 알리는 구조를 만들었음. 장애가 반복되지 않도록 구조를 바꾸는 데까지 가는 게 대응의 완성이라고 생각함.

## 🟢 Raise the Bar, Together (높은 기준, 함께)

**12. 동료의 코드나 설계에서 문제를 발견했을 때, 불편함을 감수하고 피드백한 경험이 있나요?**

동료가 설계한 파이프라인에서 watermark를 현재 시각 기준으로 찍어서 Late Arriving Data 케이스를 고려하지 않은 것을 발견했음. 직접 "이 부분 문제 있는 것 같다"고 하기보다, "이 케이스에서 어떻게 동작할 것 같아요?"라고 질문하는 방식으로 접근했음. 같이 시나리오를 짚어보면서 자연스럽게 문제를 인식하게 됐고, 수정 방향도 같이 결정했음. 지적보다 같이 생각하는 방식이 더 잘 받아들여진다는 걸 경험했음.

**13. 본인이 받은 피드백 중 처음엔 받아들이기 어려웠지만 성장에 도움이 됐던 경험을 말해주세요.**

"파이프라인이 잘 돌아가는 것만으로는 부족하다. 장애가 났을 때 다른 사람이 혼자 복구할 수 있어야 한다"는 피드백을 받았음. 처음에는 내 파이프라인이 복잡한 게 아닌데 왜 문서가 필요하냐고 생각했음. 실제로 내가 자리를 비운 날 장애가 났고 다른 사람이 원인을 파악하는 데 시간이 걸렸음. 그 이후로 알림 메시지 포맷, 재실행 절차, Known Limitation 문서화를 기본 작업으로 포함시키게 됐음.

**14. 기술적 역량이나 업무 기준이 다른 팀원과 협업할 때 어떻게 접근하나요?**

기준이 다른 팀원과 작업할 때는 내 방식을 강요하기보다 공통 기준을 먼저 맞추는 것에 집중함. 예를 들어 코드 스타일보다 "파이프라인 실패 시 알림이 가는가", "재실행 시 멱등성이 보장되는가" 같이 운영에서 꼭 필요한 최소 기준을 팀 기준으로 정했음. 세부 구현은 각자의 방식을 존중하되, 운영 기준은 공유하는 방식으로 협업했음.

**15. 팀의 기술 기준을 끌어올리기 위해 주도적으로 행동한 사례가 있나요?**

파이프라인 설계 패턴이 사람마다 달라서 장애 대응 시 다른 사람 코드를 파악하기 어려운 문제가 있었음. Airflow DAG 작성 가이드를 작성했음. Gate Task 구조, 스키마 검증 위치, 알림 포맷, watermark 관리 방법 등을 정리한 문서였음. 강제하지 않고 "이 패턴으로 짜면 장애 때 이런 이점이 있다"는 맥락과 함께 공유했음. 이후 새로운 파이프라인에서 자연스럽게 해당 패턴이 사용되는 걸 보면서 문서화의 효과를 체감했음.

## 🏢 아임웹 핏 / 동기

**16. 아임웹 도메인에서 데이터 엔지니어로 일하는 것이 어떤 매력으로 다가오나요?**

B2B SaaS이면서 B2B2C 구조라는 점이 흥미로움. 아임웹 고객(사업자)의 행동 데이터와 그 고객의 고객(쇼핑몰 이용자)의 행동 데이터가 동시에 존재하는 구조임. 단순히 한 레이어의 데이터가 아니라 계층이 있는 데이터를 설계하고 파이프라인으로 엮는 작업이 기술적으로 도전적인 문제라고 생각함. 또한 SaaS 특성상 고객 수가 늘수록 데이터 규모도 선형으로 늘어나는 구조라서, 지금부터 확장성을 고려한 파이프라인 설계가 중요한 시점이라고 판단함. 그 타이밍에 합류하고 싶음.

**17. 지금까지의 경력에서 가장 임팩트가 컸다고 생각하는 기여는 무엇인가요? 어떻게 측정했나요?**

분석팀이 원천 DB에 직접 붙어서 쿼리하던 구조를 데이터 레이크 기반으로 전환한 것임. 임팩트를 측정한 방법은 두 가지였음. 첫째, 원천 DB slow query 로그에서 분석 쿼리가 사라진 것. 둘째, 분석팀에서 "데이터가 달라요"라고 오는 문의 빈도가 줄어든 것. 수치보다 "없어진 문제"로 임팩트를 측정한 케이스였음. 데이터 엔지니어의 잘 된 일은 눈에 안 띄는 게 맞다고 생각함.

**18. 본인이 생각하는 "욕심 많은 엔지니어"란 어떤 사람인가요? 스스로 그 기준에 부합하나요?**

단순히 기술을 많이 아는 사람이 아니라, 지금 돌아가는 시스템에 만족하지 않고 더 나은 상태를 계속 그리는 사람이라고 생각함. 파이프라인이 돌아가고 있어도 "이게 장애 나면 어떻게 되지?", "스케일이 10배 되면 어디서 터지지?"를 계속 생각하는 것. 스스로는 부합하는 편이라고 생각하지만, 실행으로 이어지는 욕심인지 아닌지가 기준이라고 봄. 아이디어만 많고 실행이 없는 욕심은 팀에 도움이 안 됨.

## 🤝 CTO가 볼 수 있는 기술+문화 융합 질문

**19. 기술 부채와 빠른 실행 사이에서 팀이 올바른 방향을 유지하도록 기여할 수 있는 방식은 무엇인가요?**

기술 부채를 없애는 게 목표가 아니라, 부채를 인지하고 관리하는 게 목표라고 생각함. 당장 빠르게 쳐야 하는 작업은 Known Limitation을 명시적으로 문서화하고 언제 해소할지 시점을 잡아두는 방식을 씀. 묵히면 부채가 되지만, 관리되면 선택이 됨. 팀 차원에서는 "이건 지금 의도적으로 빠르게 간 것"과 "그냥 대충 한 것"을 구분하는 문화가 중요하다고 생각함. 그 구분이 리뷰 문화와 문서화에서 나온다고 봄.

**20. 아임웹의 데이터 플랫폼이 현재 어떤 구조일지 예상하고, 본인이 기여할 수 있는 가장 큰 영역은 어디라고 생각하나요?**

B2B SaaS 특성상 고객사별로 데이터가 분리돼 있고, 집계 레이어에서 멀티 테넌트 구조를 어떻게 다루는지가 복잡할 거라고 예상함. 원천 DB는 서비스 팀이 관리하고 데이터팀은 그걸 읽어오는 구조일 가능성이 높음. 그렇다면 스키마 변경 대응, 증분 적재 신뢰성, 레이어 간 데이터 품질 검증이 핵심 과제일 것임. 내가 기여할 수 있는 가장 큰 영역은 원천에서 분석 레이어까지의 파이프라인 안정성과 운영 가시성을 높이는 것임. 파이프라인이 언제 어디서 무슨 이유로 실패했는지 팀 전체가 빠르게 파악할 수 있는 구조를 만드는 것.
