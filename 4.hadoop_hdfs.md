# Hadoop + HDFS

---

## 아키텍처

```
Client
  ↓ (메타데이터 요청)         ↓ (데이터 읽기/쓰기)
NameNode                   DataNode × N
  - 파일 시스템 메타데이터    - 실제 블록 저장
  - 블록 위치 관리           - 블록 복제 관리
  - 파일 → 블록 매핑

SecondaryNameNode (or Standby NameNode)
  - NameNode 체크포인트 / HA Failover
```

**핵심 흐름**
1. Client가 NameNode에 파일 위치(블록 목록) 요청
2. NameNode가 DataNode 목록 반환
3. Client가 DataNode에 직접 읽기/쓰기 (NameNode를 거치지 않음)
4. DataNode들끼리 블록 복제(replication) 수행

---

## 핵심 개념

### 블록(Block)
- 기본 블록 크기: 128MB (설정 가능)
- 파일을 블록 단위로 분할해서 여러 DataNode에 분산 저장
- 대용량 파일에 최적화된 구조. 소파일(small file)이 많으면 NameNode 메모리 압박

### Replication Factor
- 기본값: 3 (같은 파일의 블록을 3개 DataNode에 복제)
- Rack Awareness: 복제본을 서로 다른 Rack에 분산해서 Rack 장애에도 내성을 가짐
  - 1번 복제본: 로컬 Rack
  - 2번 복제본: 다른 Rack
  - 3번 복제본: 다른 Rack의 다른 DataNode

### NameNode의 역할과 한계
- 모든 파일 메타데이터를 메모리에 올려서 관리함
- 파일 수가 많아질수록 NameNode 힙 메모리 사용량 증가
- **NameNode는 SPOF(Single Point of Failure)** → HA 구성 필수

NameNode가 관리하는 두 가지 파일:
- `fsimage`: 파일시스템 전체 스냅샷 (디스크 저장)
- `edits`: fsimage 이후 변경 로그 (디스크 저장, 메모리에도 반영)

기동 시 `fsimage` + `edits`를 합쳐서 메모리에 올림. edits가 너무 크면 기동 시간이 길어짐.

### Secondary NameNode
**HA의 대체가 아님.** 체크포인트 전용 프로세스임.

```
NameNode
  edits 파일이 점점 쌓임
      ↓ (주기적으로)
Secondary NameNode
  fsimage + edits를 병합 → 새 fsimage 생성
      ↓
NameNode에 새 fsimage 전달
  → edits 초기화, 기동 시간 단축
```

- Secondary NameNode는 NameNode 장애 시 자동 승격하지 않음
- 최신 edits를 갖고 있지 않아서 장애 시 일부 데이터 유실 가능
- **Non-HA 환경에서만 사용**. HA 구성 시에는 Standby NameNode가 체크포인트 역할도 겸함

### NameNode HA 구성
```
Active NameNode ←─────────────── Standby NameNode
       │                                │
       └──────── JournalNode ───────────┘
                 (edit log 공유, 3개 이상 홀수)
                        │
                   ZooKeeper
               (Active/Standby 감지 + Failover)
```

**JournalNode 역할**
- Active NameNode가 변경사항(edits)을 JournalNode에 씀
- Standby NameNode가 JournalNode에서 edits를 읽어 메모리 상태를 실시간 동기화
- Quorum 방식: 3개 중 2개 이상에 쓰면 성공. 과반수 장애 시 Active가 쓰기 중단
- JournalNode 덕분에 Standby는 항상 Active와 거의 동일한 메타데이터 상태를 유지함
- Active 장애 → ZooKeeper가 감지 → Standby가 Active로 자동 승격 (데이터 유실 없음)

**HA 모드에서 체크포인트 흐름**
HA 모드에서는 Secondary NameNode가 없음. Standby가 체크포인트 역할을 겸함.

```
Standby NameNode
  JournalNode에서 edits 계속 읽는 중
    ↓ (주기적으로)
  fsimage + edits 병합 → 새 fsimage 생성
    ↓
  새 fsimage를 Active NameNode에 전달
    → Active의 edits 파일 초기화 (무한 증가 방지)
```

승격 시 별도 동기화가 필요 없는 이유: Standby는 이미 JournalNode를 통해 실시간으로 edits를 반영하고 있었기 때문에, 승격 순간 Active와 거의 동일한 메모리 상태를 갖고 있음. 스냅샷을 새로 받을 필요 없이 바로 Active 역할을 이어받음.

Secondary NameNode와의 차이:
| | Secondary NameNode | Standby NameNode (HA) |
|---|---|---|
| 장애 시 자동 승격 | 불가 | 가능 |
| edits 동기화 | 주기적 체크포인트만 | JournalNode로 실시간 동기화 |
| 데이터 유실 | 가능 | 없음 |
| 구성 복잡도 | 단순 | 복잡 (JournalNode + ZooKeeper 필요) |

### YARN (Yet Another Resource Negotiator)
- Hadoop 2.x부터 도입된 리소스 관리 레이어
- MapReduce 외에 Spark, Tez 등 다양한 실행 엔진을 HDFS 위에서 실행 가능하게 함

```
ResourceManager  → 클러스터 전체 리소스 관리
NodeManager      → 각 노드의 컨테이너 실행/모니터링
ApplicationMaster → 개별 애플리케이션 리소스 협상
```

### 데이터 지역성 (Data Locality)
- 연산을 데이터가 있는 노드에서 실행 → 네트워크 전송 최소화
- Spark/MapReduce가 YARN에 컨테이너를 요청할 때 DataNode와 같은 노드를 우선 요청
- 클러스터가 작거나 데이터 편중이 심하면 지역성 확보가 어려움

## 실무에서 중요한 개념

### Small File Problem
- HDFS는 대용량 파일에 최적화됨. 1MB 파일 100만 개 → NameNode에 100만 개의 메타데이터 객체
- NameNode 힙 메모리: 파일 하나당 약 150byte. 1억 파일 → ~15GB 힙 필요
- **보완**: 소파일을 주기적으로 compaction하거나, 애초에 적재 시 파일 크기를 128MB 이상으로 맞춤

### 블록 크기와 병렬성의 관계
- 10GB 파일, 블록 128MB → 약 80개 블록 → 최대 80개 Mapper가 병렬 실행 가능
- 블록이 너무 크면 병렬성 저하, 너무 작으면 Small File Problem

### 복제 계수와 스토리지 비용
- Replication 3 → 실제 저장 용량은 3배
- 비용 최적화가 필요하면 Cold 데이터는 Replication 2로 낮추거나 Erasure Coding 사용
- Erasure Coding: 복제 대신 패리티 블록으로 내결함성 확보. 스토리지 효율 높지만 CPU 비용 증가

### 파일 포맷 선택
| 포맷 | 특성 | 용도 |
|---|---|---|
| Parquet | 컬럼 기반, 압축률 높음 | 분석 쿼리, Iceberg |
| ORC | 컬럼 기반, Hive 최적화 | Hive 기반 환경 |
| Avro | 행 기반, 스키마 포함 | 스트리밍, CDC |
| Text/CSV | 가독성 높음, 압축률 낮음 | Raw 적재, 디버깅 |

### Speculative Execution
- 느린 Task를 감지해서 같은 Task를 다른 노드에서 동시에 실행, 먼저 끝나는 걸 사용
- 불균형한 데이터(skew)가 있을 때 유용하지만, 부하가 2배가 될 수 있음
- 부작용이 있는 작업(외부 API 호출 등)에서는 반드시 비활성화

## 실제 운영 주의사항

### NameNode 힙 모니터링
- NameNode OOM은 클러스터 전체 장애로 이어짐
- 파일 수, 블록 수 증가 추이를 주기적으로 모니터링
- `hdfs dfsadmin -report`로 블록 수 확인
- 힙이 70% 이상이면 경고 알림 설정

### HDFS Safe Mode
- NameNode 재시작 또는 복제 블록 수가 부족할 때 자동으로 Safe Mode 진입
- Safe Mode에서는 쓰기 불가. 읽기만 가능
- `hdfs dfsadmin -safemode get`으로 상태 확인
- 정상 복제가 완료되면 자동 해제되지만, 수동 해제가 필요한 경우도 있음: `hdfs dfsadmin -safemode leave`

### DataNode 디스크 관리
- DataNode 디스크 하나가 꽉 차면 해당 노드 전체가 쓰기 불가 상태가 될 수 있음
- `dfs.datanode.failed.volumes.tolerated` 설정으로 허용 디스크 장애 수 설정
- 디스크 사용률 80% 이상이면 알림 설정

**디스크 사용률이 높을 때 대응 순서**

1. **원인 파악**: 어떤 디렉토리/테이블이 공간을 차지하는지 확인
```bash
hdfs dfs -du -h /warehouse | sort -rh | head -20
```

2. **불필요한 데이터 삭제**: TTL 지난 Raw 데이터, 임시 작업 디렉토리 정리
```bash
hdfs dfs -rm -r /tmp/spark-*
hdfs dfs -rm -r /warehouse/raw/orders/dt=2022-*
```

3. **Replication Factor 낮추기**: Cold 데이터의 복제 계수를 3→2로 줄여서 즉각 공간 확보
```bash
hdfs dfs -setrep -R 2 /warehouse/raw/orders/dt=2022-*
```

4. **Balancer 실행**: 특정 노드만 가득 찬 경우 다른 노드로 블록 재분산
```bash
hdfs balancer -threshold 10
```

5. **노드 추가**: 위 방법으로도 부족하면 DataNode 증설. NameNode 설정 없이 새 노드를 클러스터에 붙이면 자동 인식됨. 이후 Balancer 실행해서 블록 분산.

6. **Erasure Coding 전환**: 장기적으로 Cold 데이터를 Replication 대신 EC로 전환. 스토리지 효율 ~1.5배 (Replication 3 대비 절반 수준)

### Balancer 실행
- DataNode 간 블록 분포가 불균형해지면 특정 노드에 부하가 집중됨
- `hdfs balancer -threshold 10`으로 노드 간 사용률 차이를 10% 이내로 맞춤
- 트래픽이 낮은 새벽에 실행. 대역폭 제한 설정 필수 (`dfs.datanode.balance.bandwidthPerSec`)

### Trash 설정
- HDFS에서 `hadoop fs -rm`으로 삭제하면 기본적으로 `/user/<username>/.Trash`로 이동
- `fs.trash.interval` 설정으로 보존 기간 조정 (기본 0 = Trash 비활성화)
- 실수로 삭제한 데이터를 복구할 수 있는 유일한 수단. 운영 환경에서는 반드시 활성화

### 블록 손상(Corrupt Block) 대응
- `hdfs fsck / -list-corruptfileblocks`로 손상 블록 파악
- Replication이 충분하면 자동 복구됨. 복구가 안 되면 해당 파일을 Raw에서 재적재

### JournalNode / ZooKeeper 모니터링
- JournalNode 과반수 장애 시 Active NameNode가 edit log를 쓰지 못하고 중단됨
- ZooKeeper 쿼럼 장애 시 Failover가 동작하지 않음
- 두 컴포넌트 모두 헬스체크 알림 필수

### 네트워크 대역폭
- DataNode 간 복제 트래픽이 서비스 네트워크와 겹치지 않도록 분리
- 대규모 데이터 이동(Balancer, 복제 복구)이 운영 쿼리에 영향 줄 수 있음

## Q&A 심화

### Q. HDFS에서 소파일 compaction은 어떻게 하나요?

HDFS 자체에는 compaction 명령이 없음. 소파일 병합은 상위 레이어(Spark, Hive, Iceberg)에서 처리함.

**방법 1: Spark로 직접 병합**
```python
# 소파일 다수 → 파티션 재조정 후 덮어쓰기
df = spark.read.parquet("hdfs:///data/raw/orders/")
df.coalesce(10).write.mode("overwrite").parquet("hdfs:///data/raw/orders/")
```
- `coalesce`: 파티션 수를 줄임 (셔플 없음, 크기 불균형 가능)
- `repartition`: 균등 분산 (셔플 발생, 비용 있음)

**방법 2: Iceberg OPTIMIZE**
```sql
-- Iceberg가 소파일을 자동으로 병합해서 target-file-size-bytes 기준으로 재작성
CALL catalog.system.rewrite_data_files(
  table => 'db.orders',
  options => map('target-file-size-bytes', '134217728')  -- 128MB
)
```

**방법 3: Hive의 CONCATENATE**
```sql
ALTER TABLE orders PARTITION (dt='2024-01-01') CONCATENATE;
-- ORC 파일만 지원
```

실무에서는 Iceberg OPTIMIZE를 Airflow로 주기적으로 실행하는 방식이 가장 깔끔함.

### Q. 1개의 Mapper는 1개의 스레드라고 보면 되나요?

정확히는 1개의 Mapper = 1개의 JVM 프로세스(컨테이너)임.

```
YARN Container
  └── JVM 프로세스
        └── Mapper 1개 실행 (단일 스레드)
```

- Mapper는 기본적으로 단일 스레드로 동작. 멀티스레드 Mapper를 직접 구현할 수는 있지만 일반적이지 않음
- Spark는 다름. Spark Executor는 JVM 1개 안에서 여러 Task를 멀티스레드로 실행함

```
Spark Executor (JVM 프로세스)
  ├── Thread 1 → Task (= Partition) 처리
  ├── Thread 2 → Task 처리
  └── Thread N → Task 처리
      (spark.executor.cores 설정으로 스레드 수 결정)
```

MapReduce와 Spark의 차이:
| | MapReduce Mapper | Spark Task |
|---|---|---|
| 단위 | 컨테이너(JVM 프로세스) 1개 = Mapper 1개 | Executor(JVM) 1개 = Task N개 (멀티스레드) |
| 오버헤드 | JVM 기동 비용 큼 | JVM 재사용, 오버헤드 낮음 |
| 메모리 공유 | 없음 | Executor 내 스레드 간 공유 가능 |

### Q. Hadoop에서 CPU는 어디에 사용되나요?

HDFS 자체는 CPU를 거의 쓰지 않음. CPU는 주로 데이터 처리 레이어에서 사용됨.

**HDFS 레이어 (낮음)**
- NameNode: 메타데이터 처리, RPC 응답
- DataNode: 블록 읽기/쓰기, 체크섬 검증

**처리 레이어 (높음)**
| 작업 | CPU 사용 원인 |
|---|---|
| 압축/압축 해제 | Snappy, Gzip, Zstd 등 코덱 처리 |
| 직렬화/역직렬화 | Parquet, ORC, Avro 인코딩/디코딩 |
| 셔플 (MapReduce/Spark) | 정렬, 병합, 네트워크 전송 준비 |
| 집계 / 조인 | Hash aggregation, Sort-merge join |
| Erasure Coding | 패리티 블록 계산 (복제 대비 CPU 비용 증가) |
| UDF 실행 | 사용자 정의 함수 |

실무에서 CPU 병목이 생기면:
- 압축 코덱을 CPU 효율이 높은 Snappy/LZ4로 변경
- Parquet 컬럼 pruning으로 읽는 데이터 자체를 줄임
- Spark executor core 수 조정

### Q. NameNode는 메타데이터를 항상 메모리에 올려놓나요? Iceberg와의 호환성은?

**NameNode 메타데이터**
맞음. NameNode는 전체 파일시스템 메타데이터(파일 경로, 블록 목록, 권한 등)를 항상 메모리에 올려두고 운영함. 이게 빠른 응답의 이유이자, 파일 수에 따른 메모리 한계의 이유임.

**Iceberg와의 관계**
Iceberg도 자체 메타데이터 파일을 HDFS에 저장함. 구조는 아래와 같음.

```
HDFS
  /warehouse/db/orders/
    metadata/
      v1.metadata.json      ← 테이블 스냅샷, 스키마, 파티션 정보
      snap-xxx.avro         ← manifest list (어떤 manifest 파일들이 있는지)
      manifest-xxx.avro     ← manifest (어떤 데이터 파일들이 있는지)
    data/
      dt=2024-01-01/
        *.parquet           ← 실제 데이터
```

- Iceberg 메타데이터는 HDFS 파일로 저장됨 → NameNode 입장에서는 그냥 일반 파일
- Iceberg 쿼리 시 Spark/Trino가 metadata 파일을 직접 읽어서 어떤 데이터 파일을 읽을지 결정 → NameNode를 거치긴 하지만 메타데이터 파일 수는 데이터 파일보다 훨씬 적음
- Iceberg의 테이블 메타데이터(스냅샷, 스키마 등)는 Hive Metastore 또는 별도 Catalog에 저장됨. NameNode와 별개

### Q. HDFS+Iceberg vs Glue Catalog+S3+Iceberg 아키텍처 차이는?

핵심 차이는 **메타데이터를 누가 어떻게 관리하느냐**임.

```
[HDFS + Iceberg]

Client (Spark/Trino)
  ↓ 테이블 위치 조회
Hive Metastore (또는 Hadoop Catalog)
  - 테이블명 → HDFS 경로 매핑
  - 별도 서버로 운영 (MySQL/PostgreSQL 백엔드)
  ↓ 메타데이터 파일 직접 읽기
HDFS (NameNode가 파일 위치 관리)
  /warehouse/db/orders/metadata/v1.metadata.json
  /warehouse/db/orders/data/*.parquet
```

```
[Glue Catalog + S3 + Iceberg]

Client (Spark/Athena/Trino)
  ↓ 테이블 위치 조회
AWS Glue Catalog (관리형 서비스)
  - 테이블명 → S3 경로 매핑
  - NameNode 없음. Glue API로 메타데이터 조회
  ↓ 메타데이터 파일 직접 읽기
S3 (객체 스토리지)
  s3://bucket/warehouse/db/orders/metadata/v1.metadata.json
  s3://bucket/warehouse/db/orders/data/*.parquet
```

**주요 차이**

| 항목 | HDFS + Iceberg | Glue Catalog + S3 + Iceberg |
|---|---|---|
| 메타데이터 관리 | NameNode (메모리) + Hive Metastore | Glue Catalog (관리형 API) |
| 파일 위치 관리 | NameNode가 모든 파일 블록 위치를 메모리에 보관 | S3가 객체 위치 관리 (NameNode 없음) |
| 메모리 부담 | NameNode 힙에 전체 파일 메타데이터 | 없음. S3는 객체 스토리지라 별도 인덱스 서버 불필요 |
| 인프라 운영 | NameNode, DataNode, JournalNode, ZooKeeper 직접 운영 | S3, Glue는 AWS가 관리 |
| 데이터 지역성 | 가능 (연산을 DataNode에서 실행) | 불가 (S3는 원격 스토리지, 항상 네트워크 경유) |
| 비용 구조 | 서버 상시 비용 | 스토리지 + API 호출 비용 (사용량 기반) |
| 확장성 | NameNode가 병목 (파일 수 한계) | 사실상 무제한 |
| Iceberg 메타데이터 | HDFS 파일로 저장, Hive Metastore가 테이블 등록 관리 | S3 파일로 저장, Glue Catalog가 테이블 등록 관리 |

**Iceberg 관점에서 보면 둘의 차이는 작음**

Iceberg는 스토리지와 카탈로그를 추상화함. Spark 입장에서는:
```python
# HDFS 기반
spark.read.table("hive_catalog.db.orders")

# S3 + Glue 기반
spark.read.table("glue_catalog.db.orders")
```
Iceberg 레이어 위에서는 같은 방식으로 동작함. 차이는 아래 레이어(스토리지, 카탈로그)에 있음.

**언제 뭘 쓰나**
- 온프레미스 또는 기존 Hadoop 클러스터가 있으면 HDFS + Iceberg
- AWS 환경에서 새로 구축하면 S3 + Glue + Iceberg가 운영 부담이 훨씬 낮음
- 데이터 지역성이 중요한 대규모 처리(수 PB, 지연 민감)는 HDFS가 유리할 수 있음
- 그 외 대부분의 분석 워크로드는 S3 + Glue로 충분함

## 온프레미스 vs 클라우드 분산처리 플랫폼 비교

### 아키텍처 구조

```
[온프레미스]
자체 서버
  ├── HDFS (NameNode + DataNode)
  ├── YARN (ResourceManager + NodeManager)
  ├── Spark / MapReduce
  └── Hive Metastore

[클라우드 (AWS 기준)]
  ├── S3 (스토리지)
  ├── Glue Catalog (메타데이터)
  ├── EMR (Spark 실행, 필요할 때만 클러스터 기동)
  └── Athena (서버리스 쿼리)
```

---

### 장단점 비교

| 항목 | 온프레미스 (HDFS) | 클라우드 (S3 + EMR) |
|---|---|---|
| **초기 비용** | 서버 구매 비용 큼 | 없음 (사용량 기반) |
| **운영 부담** | NameNode, DataNode, ZooKeeper 등 직접 관리 | AWS가 인프라 관리 |
| **확장성** | 노드 증설 필요 (시간/비용 소요) | 수분 내 클러스터 확장 가능 |
| **성능** | Data Locality로 로컬 디스크 읽기 | 항상 네트워크 경유 (단, AWS 내부망은 빠름) |
| **비용 예측** | 고정 비용 (서버 상시 운영) | 변동 비용 (사용량에 따라 급등 가능) |
| **데이터 주권** | 사내 보관, 외부 유출 없음 | 클라우드 벤더에 데이터 위탁 |
| **보안/컴플라이언스** | 직접 통제 가능 | 클라우드 보안 정책에 의존 |
| **클러스터 유연성** | 상시 기동 (유휴 자원 낭비) | 필요할 때만 기동 후 종료 가능 |
| **장애 대응** | 직접 대응 필요 | AWS SLA 내에서 자동 복구 |

---

### 비용 구조 차이

**온프레미스**
```
서버 구매 (초기 대규모 투자)
  + 전기/냉각 비용 (상시)
  + 인건비 (클러스터 운영 엔지니어)
  + 유휴 자원 비용 (피크 대비 서버 확보 → 평상시 낭비)
```

**클라우드**
```
S3 스토리지 비용 (GB당 과금)
  + EMR 클러스터 비용 (실행 시간당 과금)
  + 네트워크 비용 (데이터 전송량)
  → 피크 시 자동 확장, 작업 끝나면 클러스터 종료
  → 단, 상시 분석 워크로드가 많으면 온프레미스보다 비쌀 수 있음
```

---

### 유즈케이스별 선택 기준

**온프레미스가 유리한 경우**
- 데이터 규모가 수 PB 이상이고 상시 대규모 처리가 필요한 경우
  - 클라우드 비용이 온프레미스 서버 비용을 넘어서는 시점
- 금융, 의료 등 데이터 외부 반출이 법적으로 제한되는 경우
- 네트워크 지연에 극도로 민감한 실시간 처리
- 기존 Hadoop 클러스터 투자가 이미 되어 있는 경우 (마이그레이션 비용 > 클라우드 이점)

**클라우드가 유리한 경우**
- 배치 작업이 불규칙적이거나 야간에만 돌리는 경우 (EMR 클러스터를 작업 시간에만 켬)
- 초기 스타트업 또는 데이터 규모가 빠르게 변하는 경우 (확장/축소 유연성)
- 데이터 엔지니어링 팀이 작아서 인프라 운영 리소스가 부족한 경우
- 분석/리포팅 위주 워크로드 (실시간성 불필요, Athena로 서버리스 처리)
- 멀티 리전 데이터 복제가 필요한 경우

**혼합(Hybrid) 구조**
- 온프레미스에서 원천 적재 후 클라우드로 동기화
- 민감 데이터는 온프레미스, 분석용 집계 데이터는 클라우드
- 대규모 배치는 온프레미스, 탄력적 피크 처리는 클라우드 Burst

---

### 결론

새로 구축하는 경우라면 특별한 이유(데이터 주권, 수 PB 규모, 컴플라이언스)가 없으면 클라우드가 현실적으로 나은 선택임. 운영 부담 차이가 너무 큼.

온프레미스는 이미 투자가 되어 있거나, 클라우드 비용이 역전되는 규모가 되거나, 법적 제약이 있을 때 선택하는 방식임.
