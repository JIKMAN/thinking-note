# Q. 데이터 레이크 계층에서 스키마 관리를 어떻게 하는가?

Iceberg 기반으로 Schemaless로 적재하되 변경을 감지하고 통제하는 구조로 설계함.
엄격한 통제는 신뢰도가 높지만 유연함이 부족하기 때문에, 레이어별로 허용 수준을 다르게 가져가는 방식을 씀.

---

## 레이어 구조

```
원천
 ↓
Raw Zone     → Schemaless, 원천 원형 보존, mergeSchema=true
 ↓
Staged Zone  → 컬럼 추가만 허용, 타입 변경은 차단
 ↓
Curated Zone → 타입 강제, NOT NULL, 비즈니스 규칙 적용
```

---

## 스키마 변경 감지 및 대응 (Airflow DAG 기준)

| 변경 유형 | 대응 |
|---|---|
| 컬럼 추가 | Slack 알림 + 적재 허용 (nullable이라 안전) |
| 컬럼 삭제 | Slack 알림 + 수동 확인 요청 |
| 타입 변경 | 파이프라인 중단 + DLQ 라우팅 |

필요시 `meta.schema_changelog`에 히스토리 기록함.

---

## 케이스별 흐름

**1. 평상시**
Gate 통과 → 그냥 적재 → 아무 알림 없음

**2. 컬럼 추가 감지**
Slack 알림: `"orders에 new_col(STRING) 추가됨"`
→ 자동 적재 허용
→ changelog INSERT
→ 담당자가 나중에 확인 후 다운스트림 반영 여부 결정

**3. 타입 변경 / 컬럼 삭제 감지**
Slack 알림: `"🚨 orders.amount 타입 변경 INT→STRING"`
→ DAG 실패 처리 (적재 안 함)
→ 담당자가 원천 확인 후 수동으로 승인하면 재실행

---

## Q. 왜 Strict 스키마를 DI에 적용하지 않는가?

### 원천은 내가 통제할 수 없음
원천 DB는 보통 다른 팀, 다른 서비스가 관리함. 그쪽에서 컬럼 하나 추가했을 때 내 파이프라인이 터지면, 원천 변경 → 내 파이프라인 수정 → 배포 사이클이 생김. 원천 팀 배포 속도에 내 파이프라인이 종속되는 구조가 됨.

### 컬럼 추가는 사실 위험하지 않음
Iceberg 기준으로 컬럼이 추가되면 기존 파티션에서는 NULL로 읽힘. 다운스트림이 `SELECT *`를 쓰지 않는 한, 기존 쿼리는 그냥 동작함. 이걸 파이프라인 중단으로 막는 건 과잉 대응임.

### Strict 스키마의 비용
- Schema Registry 운영 비용(오버스펙)
- 원천 변경마다 스키마 버전 올리고 호환성 체크
- 파이프라인 배포 사이클 증가
- 원천 팀과의 조율 비용

이 비용을 DI 레이어에서 지불하면서 얻는 건 사실상 "잘못된 데이터를 조금 더 빨리 인지하는 것" 뿐임.

### DI의 데이터는 재처리가 가능함
원천을 Raw에 보존해두기 때문에, 스키마 변경이 나중에 발견되더라도 해당 시점부터 재처리하면 됨. 원천이 사라지는 게 아니니까 돌이킬 수 없는 손실이 아님.

### 감지 시점이 충분히 빠름
인제스트 앞단 Gate Task에서 잡으면 배치 단위로 감지됨. 문제가 있는 데이터가 Curated까지 올라가기 전에 차단되는 구조면, 비즈니스 영향이 발생하기 전에 대응 가능함.
