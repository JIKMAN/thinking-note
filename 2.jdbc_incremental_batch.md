# Q. JDBC + updated_at 기반 증분 배치의 약점과 보완 방법은?

binlog CDC가 아닌 JDBC + `updated_at` 기반 증분 배치를 쓰고 있음.
이 방식은 구현이 단순하고 원천 DB 의존성이 낮지만, 구조적으로 놓치는 케이스가 존재함.

---

## 핵심 약점 요약

| 약점 | 원인 | 보완 |
|---|---|---|
| Hard Delete 감지 불가 | 삭제된 행은 `updated_at`이 없음 | Soft Delete 강제화 또는 전체 스캔 차집합 |
| Late Arriving Data | 트랜잭션 커밋 지연, 복제 지연 | watermark에 overlap 버퍼 적용 |
| updated_at 미갱신 | bulk update, DBA 직접 수정 | 보완 불가 케이스로 인지, 모니터링으로 감지 |
| 원천 DB 직접 부하 | JDBC가 원천 서비스 DB에 붙음 | Read Replica 사용 |
| Exactly-once 미보장 | 실패 후 재실행 시 중복 가능 | Iceberg MERGE INTO로 idempotent upsert |
| 긴 트랜잭션 경계 무시 | 시간 기준 슬라이싱이라 커밋 타이밍 이슈 | watermark overlap으로 완화 |

---

## Q. 삭제된 데이터는 어떻게 감지하나요?

`updated_at` 방식의 가장 큰 구조적 한계임. Hard Delete는 원천에서 행이 사라지기 때문에 증분 배치로는 감지 자체가 불가능함.

**보완 방법**
1. **Soft Delete 강제화**: 원천 팀과 협의해서 `is_deleted`, `deleted_at` 컬럼으로 삭제를 표현하도록 함. 삭제가 UPDATE로 처리되므로 `updated_at`으로 감지 가능해짐.
2. **전체 스캔 차집합**: 주기적(예: 매일 새벽)으로 원천 전체 PK를 풀스캔해서 Raw에 있는 PK와 차집합을 구함. 비용이 크기 때문에 자주 돌리기 어렵고, 삭제 감지 지연이 발생함.
3. **포기하고 명시**: 삭제가 비즈니스적으로 의미 없거나 빈도가 낮으면, 이 케이스를 명시적으로 Known Limitation으로 문서화하고 다운스트림에 공지함.

현실적으로는 Soft Delete 강제화가 가장 깔끔함. 단, 원천 팀의 협조가 전제됨.

---

## Q. updated_at이 항상 갱신된다고 보장할 수 있나요?

보장 못 함. 대표적인 누락 케이스:

- **Bulk UPDATE**: `UPDATE orders SET status = 'CLOSED' WHERE ...` 처럼 애플리케이션 바깥에서 직접 실행하면 `updated_at` 트리거가 없는 경우 미갱신
- **DBA 직접 수정**: 장애 대응이나 데이터 픽스 작업 시 ORM을 거치지 않음
- **애플리케이션 버그**: `updated_at`을 application 레벨에서 관리할 경우 누락 가능

**보완 방법**
- DB 트리거로 `updated_at`을 강제 갱신하도록 원천 팀에 요청함. 애플리케이션 레벨 의존보다 훨씬 안전함.
- 트리거도 없고 협의도 안 되면, 해당 테이블은 Known Limitation으로 분류하고 정기 전체 스캔을 병행함.

완전한 보완은 어렵고, 이 한계가 있다는 걸 알고 설계에 반영하는 게 현실적임.

## Q. 배치 실행 중에 들어온 데이터는 어떻게 처리하나요? (Watermark 경계 문제)

배치 시작 시점에 `MAX(updated_at)`을 watermark로 찍고 `updated_at > last_watermark` 조건으로 가져옴. 이때 배치 실행 중에 커밋된 데이터는 다음 배치에서 가져가야 하는데, 시간 기준이다 보니 경계가 불명확함.

**보완 방법**
- watermark를 `현재 시각 - N분`으로 설정해서 실행 중 커밋된 데이터를 다음 배치에서 확실히 가져가도록 함
- `>=` 조건으로 중복 포함 후 Iceberg MERGE INTO로 upsert 처리하면, 중복이 들어와도 덮어써지므로 문제 없음
- 결국 idempotent upsert 구조가 있어야 이 문제를 안전하게 처리 가능함

---

## Q. 늦게 도착한 데이터(Late Arriving Data)는 어떻게 처리하나요?

트랜잭션이 길거나 Read Replica를 쓰는 경우 복제 지연이 발생함. 예를 들어 `updated_at = 10:00`인 데이터가 실제로 Read Replica에 반영되는 시점이 10:03일 수 있음. watermark를 10:00으로 찍고 배치를 10:01에 실행하면 이 데이터를 놓침.

**보완 방법**
- watermark를 `현재 시각 - 버퍼(예: 5~10분)`으로 설정함. 최근 N분은 항상 overlap해서 재처리함
- overlap 구간의 중복은 Iceberg MERGE INTO로 흡수함
- 버퍼 크기는 원천 DB의 복제 지연 SLA 기준으로 설정함

---

## Q. 원천 DB에 직접 붙는 건 부하 걱정이 없나요?

걱정이 맞음. 원천 서비스의 Primary DB에 직접 JDBC로 붙으면 대용량 테이블 풀스캔 시 원천 서비스에 영향을 줄 수 있음.

**보완 방법**
- **Read Replica 사용**: 원천 Primary가 아닌 Read Replica에서 읽음. 원천 서비스 트래픽과 분리됨.
- **배치 시간 조정**: 원천 서비스 트래픽이 낮은 새벽 시간대에 실행함.
- **`updated_at` 인덱스 확인**: 인덱스가 없으면 풀스캔이 발생함. 원천 팀에 인덱스 추가를 요청하거나, 없으면 조회 비용을 감수하고 설계함.

---

## Q. 배치 실패 후 재실행하면 중복 데이터가 들어오지 않나요? (Exactly-once)

JDBC 증분 배치는 구조상 at-least-once임. 실패 후 watermark를 되돌려서 재실행하면 같은 데이터를 다시 가져오게 됨.

**보완 방법**
- Iceberg `MERGE INTO` (upsert)로 적재함. PK 기준으로 이미 있는 행은 UPDATE, 없는 행은 INSERT 처리되므로 중복 행이 생기지 않음.
- Raw Zone에도 중복 행이 쌓이는 게 싫다면, Raw는 append-only로 두고 Staged 이상에서 dedup 처리함.
- watermark를 메타 테이블에 저장하고, 배치 성공 시에만 커밋함. 실패 시 이전 watermark에서 재실행되도록 보장함.

---

## Q. CDC 대비 지연은 얼마나 허용 가능한가요? 지연이 문제가 된 적 없나요?

현재 구조는 배치 주기만큼의 지연이 발생함. 예를 들어 1시간 배치면 최대 1시간 지연.

- 실시간 대시보드나 이벤트 기반 처리가 필요한 경우 이 방식은 맞지 않음
- 분석/리포팅 용도라면 수십 분~수 시간 지연은 보통 허용 가능함
- 지연이 문제가 된다면 배치 주기를 줄이거나, 해당 테이블만 binlog CDC로 전환하는 선택지를 검토함

현재 유스케이스가 분석 목적이라 배치 지연이 허용 가능한 범위 내에 있음.

---

## Q. updated_at도 없는 테이블은 어떻게 처리하나요?

이 경우는 증분 배치 자체가 불가능함. 선택지는 세 가지임.

1. **전체 재적재(Full Load)**: 매번 전체를 다시 가져옴. 테이블이 크면 비용이 큼.
2. **PK 기반 차집합**: 마지막으로 본 MAX(PK)보다 큰 행만 가져옴. 단, INSERT-only 테이블에서만 유효함. UPDATE는 감지 못 함.
3. **원천 팀에 컬럼 추가 요청**: `updated_at`, `created_at` 컬럼을 추가해달라고 요청함.

현실적으로는 테이블 규모와 변경 빈도에 따라 Full Load vs PK 기반을 선택함.

---

## Q. 배치 실패 후 재처리 시 그 사이에 UPDATE된 데이터는 어떻게 되나요?

누락됨. 이게 JDBC 증분 배치의 구조적 한계 중 하나임.

**시나리오**
```
10:00 ~ 11:00 배치 실행 → 실패
  → 이 시점에 orders.id=1 의 updated_at = 10:30 (범위 내)

재처리 시점: 11:30
  → 그 사이 orders.id=1 이 11:10에 UPDATE됨
  → updated_at = 11:10 으로 바뀜

재처리 범위: updated_at > 10:00 AND <= 11:00
  → orders.id=1 은 이제 11:10이라 범위 밖 → 누락
```

최신 상태(11:10 버전)는 다음 배치(11:00~12:00)에서 들어오지만, 중간 상태(10:30 버전)는 영원히 못 가져옴.

### 왜 구조적 한계인가

JDBC 증분 배치는 본질적으로 "현재 상태 스냅샷"을 찍는 방식임. 이벤트를 기록하는 게 아니라 그 시점의 행 상태를 읽는 거라, 중간에 여러 번 바뀐 상태는 원래부터 보장 안 됨.

binlog CDC는 이 문제가 없음. Kafka에 10:30 UPDATE 이벤트, 11:10 UPDATE 이벤트가 순서대로 남아있으니까.

### 실무에서의 판단

- 중간 상태가 필요 없고 최종 상태만 맞으면 되는 유스케이스(분석, 리포팅)라면 허용 가능한 트레이드오프임
- 이벤트 히스토리나 감사 로그가 필요한 경우라면 이 방식은 근본적으로 맞지 않음 → CDC가 맞는 선택

"맞습니다. 단순히 MERGE INTO를 걸면 타겟 테이블 전체를 스캔하며 비교하기 때문에 성능이 기하급수적으로 떨어집니다. 이를 방지하기 위해 타겟 Iceberg 테이블에도 updated_at 기반의 파티션(예: Day 또는 Month)을 설정합니다. 그리고 MERGE 조건절(ON)에 source.id = target.id 뿐만 아니라, AND target.updated_at >= [버퍼 시간의 파티션 범위]처럼 명시적인 파티션 필터를 주입합니다. 이를 통해 Iceberg 엔진이 전체 데이터 파일 대신 **가장 최근에 변경된 데이터가 속한 파티션 파일들만 스캔하여 병합(Partition Pruning)**하도록 스캔 범위를 극적으로 좁혀서 비용 문제를 해결하고 있습니다."

(예: 1시간 안에 '결제 성공 -> 취소 -> 환불' 여러 번 바뀐 로그가 필요할 때). JDBC 배치 구조의 한계인 Intermediate State 누락이 허용되지 않는 시점이 찾아오면, 가장 핵심 도메인 테이블부터 점진적으로 CDC 스트리밍 아키텍처로 전환을 시작할 것입니다."
