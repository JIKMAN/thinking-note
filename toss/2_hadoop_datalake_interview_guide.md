# 2. 하둡 에코시스템 및 최신 Data Lake (Iceberg, HDFS, Hive)

## 📌 나의 경험 (이력서 기반)
- HDFS, Hive 기반 온프레미스 Data Lake / DW 운영 경험 (줌인터넷)
- Apache Iceberg 도입으로 대규모 데이터 업데이트 및 정적 품질 확보 (트래블월렛, 줌인터넷)
- 기존 온프레미스 하둡 환경에서 AWS 클라우드 데이터 레이크(S3, Glue, Iceberg)로의 아키텍처 마이그레이션 수행
- 트랜잭션 기반 통합 저장 구조 설계

## 🎯 토스뱅크 요구 역량 (JD 매핑)
- "Hadoop Ecosystem 환경 기반의 개발에 능숙하신 분"
- "데이터 플랫폼 고도화를 위해 Iceberg, Trino 같은 새로운 기술을 검토하고 도입"

## 💡 핵심 어필 포인트
전통적인 하둡(HDFS/Hive) 생태계를 경험해본 것뿐만 아니라, 그 한계를 느껴 최신의 Iceberg+클라우드 조합으로 성공적인 마이그레이션을 이뤄낸 과정이 매우 큰 경쟁력입니다. `왜 Iceberg였는가?` 에 대한 기술적 고찰과 도입 결단을 명확히 설명해야 합니다.

---

## 📝 예상 면접 질문 및 답변 준비 가이드

**Q1. 기존 온프레미스 하둡(Hive/HDFS) 환경에서 AWS 기반 Iceberg 데이터 레이크로 마이그레이션하게 된 결정적인 기술적 배경은 무엇인가요?**
* **가이드:** 하둡이나 Hive가 가지는 파일 단위 락(Lock) 문제, 스키마 에볼루션의 어려움, S3(Object Storage)와 Hive의 호환성 문제 등을 언급하며 Iceberg가 이를 어떻게 해결했는지 어필하세요.

**Q2. Apache Iceberg를 도입할 때 겪었던 어려움이나 성능 이슈, 그리고 이를 어떻게 해결하셨는지 궁금합니다.**
* **가이드:** Iceberg의 메타데이터 비대화(Metadata Bloat) 관련 Compaction 문제, 쿼리 플래닝 지연 이슈 등에 대한 실무적 트러블슈팅 경험을 묻습니다. Maintenance 작업(Data/Metadata Compaction, Expire Snapshots) 등의 운영 자동화 경험을 제시하세요.

**Q3. 온프레미스와 AWS 클라우드 환경의 데이터 인프라를 모두 운영해 보셨는데, 두 가지 인프라를 운영할 때 각각 어떤 부분에 중점을 두셨나요?**
* **가이드:** 온프레미스는 자원의 집약적 관리 및 하드웨어적 안정성, 클라우드는 자원 탄력성(Scale-out)과 비용 관리(Cost Optimization) 관점에서의 차이를 논리적으로 답변하세요.

**Q4. 대규모 업데이트 발생 시 정합성을 어떻게 확보하셨으며, 트랜잭션 기반 통합 저장 구조 설계 중 가장 신경 쓴 설계 원칙은 무엇인가요?**
* **가이드:** 금융권 데이터에 준하는 결제, 정산, 유저 데이터 등의 트랜잭션 빈도를 어떻게 Iceberg 로 무리 없이 소화했는지 기술하세요.
