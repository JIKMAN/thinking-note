# 5. 오픈소스 분석 플랫폼 고도화 및 운영 (ClickHouse, Trino, BI)

## 📌 나의 경험 (이력서 기반)
- 실시간 집계를 위한 ClickHouse 클러스터 도입 및 Materialized View 기반 실시간 데이터 집계 설계
- Superset 연결, 전사 실무 조직 역량 지원을 위한 실시간 분석/BI 대시보드 환경 구축 (트래블월렛)
- 타겟팅 유저 피처 제공(서빙) 시스템 구축, HBase 활용 실시간 서빙.

## 🎯 토스뱅크 요구 역량 (JD 매핑)
- "오픈소스 분석 플랫폼의 성능 및 안정성 최적화를 담당"
- "분석 플랫폼 고도화를 위해 새로운 기술(Trino 등)을 검토하고 버전 조합 관리"
- "직접 오픈소스 소스 코드를 분석하고 패치/백포팅 등의 기여 경험 우대"

## 💡 핵심 어필 포인트
토스뱅크는 데이터 제공의 레이턴시(Latency)를 극단적으로 낮추고, 최신의 OLAP/서빙 기술을 활용하려는 조직입니다. 이력의 `ClickHouse 도입 및 최적화` 경험이 정확히 맞아떨어집니다. 왜 ClickHouse를 썼고, 그 안에서 성능을 극대화하기 위해 `어떤 기술 검토와 리소스 관리`를 했는지 강조하세요.

---

## 📝 예상 면접 질문 및 답변 준비 가이드

**Q1. 다양한 OLAP 또는 분산 쿼리 엔진(예: Trino(Presto), Druid 등) 중 실시간 분석 환경으로 ClickHouse를 최종 결정하게 된 기술검토 과정은 어떠했나요?**
* **가이드:** 기술 스택 선택의 당위성을 묻는 질문입니다. 타 엔진에 비해 ClickHouse가 가진 장점(Columnar Storage, Vectorized execution 등)과 현업의 초당 쿼리 요구량, 집계 딜레이 요구사항을 비교해 설명하세요.

**Q2. ClickHouse 혹은 다른 엔진을 도입·최적화하시면서 오픈소스의 버그를 마주하거나 문서 범위를 벗어나는 파츠가 있었을 것으로 생각되는데, 소스코드를 직접 분석하거나 관련 이슈를 파고든 경험이 있나요?**
* **가이드:** JD에서 '직접 소스 코드를 분석하고 패치하는 기여 경험'을 우대합니다. (반드시 코드 컨트리뷰션이 아니더라도) 문제 해결을 위해 오픈소스의 Github Issue 탭, 커뮤니티, 혹은 내부 엔진 구조(C++ 등)까지 확인해서 우회(Workaround) 방법을 찾아낸 경험을 필살기로 준비하세요.

**Q3. Materialized View를 구축할 때 뷰 업데이트에 따른 디비 부하나 데이터 변경(Update/Delete)에 취약한 점을 어떻게 커버하셨는지요?**
* **가이드:** ClickHouse(혹은 일반 OLAP 엔진)의 아키텍처 이해도를 묻는 질문입니다. 과도한 Join 최소화 전략, 백그라운드 머지(Merge) 로직 등에 대해 설명하여 깊이 있는 최적화 경험을 보여주세요.

**Q4. (HBase 등 서빙 시스템 관련) 서버리스 데이터 API 서빙 환경을 구축하셨는데, 수많은 Ad-hoc 권한이 들어올 때 초당 쿼리 부하(QPS) 관리와 안정성 확보는 어떻게 진행하셨나요?**
* **가이드:** 실시간 서빙의 장애 가능성, 캐싱 아키텍처 설계, 혹은 인프라 리소스 증설 등에 관한 아키텍처 노하우를 물어보는 질문입니다. 사용자 체감 성능 향상을 수치적으로 언급하세요.
