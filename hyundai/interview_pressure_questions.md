## PART 1. 아키텍처 설계 (PT Slide 3 — 가장 중요)

### Q1. PT에서 Lakehouse(Iceberg+S3)와 ClickHouse OLAP을 이중으로 운영하는 아키텍처를 선택하셨는데, 왜 하나로 통합하지 않고 두 레이어를 유지했나요?

🟢 광고 플랫폼 프로젝트 진행 과정에서, 광고 로우 데이터와 유저 피처 데이터를 혼합한 실시간 집계 테이블에 대한 요구사항이 생겼습니다. 100ms 수준의 쿼리 응답속도가 요구됐는데, 데이터 레이크 기반의 Athena나 Spark로는 이 레이턴시를 만족할 수 없었습니다. 반면 ClickHouse는 이미 집계된 Materialized View를 통해 해당 요구사항을 충족할 수 있었기 때문에, 대용량 배치 처리와 이력 보관은 Iceberg Lakehouse가, 실시간 집계 서빙은 ClickHouse OLAP이 담당하는 이중 구조로 설계하게 됐습니다.

**꼬리질문**

- 1-1. ClickHouse만으로 전체 플랫폼을 구성하면 안 되는 이유가 뭔가요? ClickHouse도 S3-backed 스토리지(ClickHouse Cloud)를 지원하는데.

🔧 **[기존 답변 수정]** 기존 답변에서 "ClickHouse는 수직 확장만 가능하다"고 하셨는데, ClickHouse는 샤딩을 통한 수평 확장도 지원합니다. 실제 차별점을 정확하게 말씀하셔야 합니다.

> ClickHouse는 OLAP 특화 엔진이기 때문에 ACID 트랜잭션, 스키마 진화, Time Travel, 파티션 단위 overwrite 같은 데이터 레이크하우스 기능이 없습니다. 결제·정산·AML 데이터처럼 이벤트 단위 정합성 보장과 재처리가 필요한 도메인은 Iceberg가 필수적이었습니다. 또한 Iceberg는 S3에 저장하기 때문에 장기 보관 비용이 ClickHouse 대비 훨씬 저렴합니다. ClickHouse만으로 운영하면 저장 비용과 트랜잭션 보장 측면에서 한계가 있었습니다.

- 1-2. 반대로 Iceberg + Trino/Athena 조합만으로 실시간 요구사항을 만족시킬 수는 없었나요? 왜 별도 OLAP 레이어가 필요했나요?

🟢 Athena는 S3에서 파일을 스캔하는 구조라 콜드 쿼리 기준 수 초~수십 초 레이턴시가 발생합니다. 광고 백엔드 API에서 실시간으로 모수를 조회하는 요구사항(100ms 이내)에는 물리적으로 대응이 불가능했습니다. Iceberg + Athena는 배치 분석과 Ad-hoc 쿼리에는 충분하지만, 저지연 서빙 레이어로는 적합하지 않아 ClickHouse를 별도 OLAP 레이어로 도입했습니다.

- 1-3. 두 시스템 간 데이터 동기화 지연이 발생하면 어떻게 됩니까? Iceberg Gold Layer와 ClickHouse Aggregation Table이 불일치하는 상황이 실제로 있었나요? 어떻게 탐지하고 해결했나요?

🟢 실제로 초기 도입 시에 Iceberg 테이블에 MOR 방식으로 데이터를 적재할 때 ClickHouse Dictionary가 이를 동기화하지 못하는 문제가 있었습니다. 문제는 Dictionary가 별도 에러 없이 이전 스냅샷 데이터를 조용히 반환하고 있어서 탐지가 늦었습니다. 이후 Iceberg와 ClickHouse의 주요 집계값(COUNT, SUM)을 주기적으로 비교하는 모니터링 DAG를 별도로 구축했고, 임계치 이상 불일치 발생 시 Slack 알림을 발송하도록 했습니다. 구조적으로는 Dictionary와 동기화하는 테이블은 COW 모드만 사용하도록 표준화하여 재발을 방지했습니다.

- 1-4. 이 이중 구조를 유지하는 운영 비용(인프라비 + 엔지니어링 리소스)이 단일 구조 대비 얼마나 더 들었는지 추정해보셨나요?

🟢 초기 ClickHouse 설계 시 EFS 기반 스토리지를 사용했는데, IO 작업에서 발생하는 비용이 과도하게 컸고 이를 EBS로 전환하면서 스토리지 비용을 50% 이상 절감했습니다. 또한 실시간 적재 시 발생하는 small parts 문제로 인해 백그라운드 merge IO 비용이 과도하게 발생했는데, 앞단에 Buffer 엔진을 두어 write 빈도를 줄이는 튜닝으로 이 비용도 줄였습니다. 엔지니어링 리소스 측면에서는 두 시스템 각각의 운영 부담이 있지만, ClickHouse는 한번 구조가 잡히면 운영 오버헤드가 크지 않아 1인 기준으로도 감당 가능한 수준이었습니다.

### Q2. 아키텍처 다이어그램을 보면 Kafka → Spark Streaming → Buffer Engine → Raw Data Tables → Materialized View 흐름인데, Buffer Engine의 역할을 구체적으로 설명해주세요.

🟢 ClickHouse는 데이터를 삽입할 때마다 디스크에 파트(part)를 생성하고, 백그라운드에서 이를 병합합니다. 삽입 빈도가 높으면 small parts가 과도하게 쌓여 "too many parts" 에러가 발생하고 백그라운드 merge IO 비용이 급증합니다. Buffer 엔진은 메모리에 데이터를 일시 보관했다가 설정한 임계치(행 수, 바이트, 시간)에 도달하면 한 번에 타겟 테이블로 flush하는 역할을 합니다. 이를 통해 insert 빈도를 낮추고 파트 생성 수를 줄여 ClickHouse의 안정성과 성능을 확보했습니다.

**꼬리질문**

- 2-1. ClickHouse Buffer Table은 메모리 기반인데, 장애 시 데이터 유실 리스크가 있습니다. 이를 어떻게 보완하셨나요?

🟢 ClickHouse에 적재되는 로우 데이터는 Iceberg에도 동일하게 병렬 적재하고 있습니다. 따라서 Buffer 장애로 유실이 발생해도 Iceberg에서 해당 시간대 데이터를 재적재하는 복구 스크립트로 대응합니다. 또한 ClickHouse 클러스터 OOM이나 장애 시 실시간 집계에도 즉각 영향이 생기기 때문에, 해당 알림은 긴급도를 높여 모니터링하고 있습니다.

- 2-2. Buffer Table의 flush 주기(min/max rows, bytes)를 어떻게 설정하셨나요? 그 설정값의 근거는?

🔧 **[기존 답변 보완]** "1시간으로 적용"만 언급하셨는데, Buffer Table은 시간 외에도 행 수·바이트 기준도 함께 설정해야 합니다.

> 원천 데이터가 배치성이기 때문에 max_time을 배치 주기에 맞춰 3600초(1시간)로 설정했습니다. 동시에 max_rows는 메모리 과부하 방지를 위해 100만 건, max_bytes는 512MB로 설정하여 시간 이전에도 임계치 도달 시 flush되도록 했습니다. min 값은 너무 잦은 flush를 방지하기 위해 min_time=600초, min_rows=10만 건으로 설정했습니다.

- 2-3. Spark Streaming에서 ClickHouse로 insert 시 backpressure가 발생하면 어떻게 처리하셨나요?

🟢 Kafka에서 읽어오는 마이크로 배치당 레코드 수를 `maxOffsetsPerTrigger` 옵션으로 제한하여 배압을 적용하고 있습니다. 이렇게 하면 ClickHouse의 처리 속도보다 Spark가 빠르게 insert를 쏟아내는 상황을 방지할 수 있습니다.

### Q3. PT 아키텍처에 "Dictionary Engine - memory table"이 있는데, 이게 무엇이고 왜 필요했나요?

✅ ClickHouse Dictionary는 외부 소스(S3, RDS, Iceberg 등)에서 데이터를 읽어 ClickHouse 메모리에 key-value 형태로 상주시키는 인메모리 조회 테이블입니다. PT 아키텍처에서는 유저 정보나 캠페인 메타 같은 디멘션 데이터를 Dictionary로 올려두고, 광고 이벤트 집계 시 Materialized View에서 Dictionary lookup으로 조인하는 구조로 활용했습니다. 일반적인 JOIN보다 훨씬 빠르고, Materialized View가 행 단위로 실시간 처리할 때 분산 조인 없이 로컬에서 조회가 가능하다는 것이 핵심 이점입니다.

**꼬리질문**

- 3-1. Dictionary를 사용한 join이 일반 JOIN과 비교해서 어떤 성능 이점이 있나요?

🔧 **[기존 답변 수정]** 기존 답변은 "Iceberg 테이블을 읽는 편리함"을 말씀하셨는데, 이는 Dictionary JOIN의 핵심 이점이 아닙니다.

> 일반 JOIN은 두 테이블을 메모리에 올려 셔플 또는 해시 조인을 수행하는데, 분산 환경에서는 네트워크 통신이 발생합니다. Dictionary는 이미 각 ClickHouse 노드의 메모리에 올라가 있는 해시맵이기 때문에, lookupJoin 또는 dictGet() 함수로 O(1) 로컬 해시 조회가 가능합니다. 특히 Materialized View처럼 행이 insert될 때마다 즉각 실행되는 환경에서는 네트워크 왕복이 없는 인메모리 조회가 성능에 결정적 차이를 만듭니다.

- 3-2. Dictionary의 데이터가 stale해지면 어떻게 됩니까? refresh 전략은?

🔧 **[기존 답변 작성]** "stale이 뭐임??"으로 남겨두셨는데 반드시 알아야 하는 개념입니다.

> Stale이란 Dictionary의 소스 데이터가 업데이트됐지만 Dictionary가 아직 반영하지 못한 상태, 즉 '만료된 데이터'를 말합니다. Dictionary는 `LIFETIME` 설정으로 자동 갱신 주기를 정의합니다. 예를 들어 `LIFETIME(MIN 300 MAX 3600)`으로 설정하면 5~60분 사이 랜덤하게 전체 재로드합니다. 실제로 운영에서는 유저 정보처럼 업데이트 빈도가 낮은 데이터는 LIFETIME을 길게, 캠페인 상태처럼 자주 바뀌는 데이터는 짧게 설정했습니다. Stale 기간 동안에는 이전 값이 반환되기 때문에, 실시간 정합성이 중요한 테이블은 Dictionary 대신 직접 JOIN을 사용하는 것이 맞습니다.

- 3-3. Dictionary로 관리하기에 데이터가 너무 커지는 경우 어떻게 처리하셨나요?

🟢 Dictionary는 ClickHouse 메모리에 전체 데이터가 올라가야 하기 때문에 메모리 한계를 초과하는 대용량 테이블에는 적합하지 않습니다. 실제로 Dictionary 적용 대상을 수십만 건 이하의 디멘션 테이블로 제한하고, 그 이상이 되는 경우에는 JOIN으로 처리하거나, Dictionary의 `complex_key_hashed` 레이아웃과 메모리 한도를 설정해 OOM을 방지하고 있습니다.

### Q4. Bronze → Silver → Gold 레이어를 어떤 기준으로 데이터를 나눴나요? 각 레이어 간 SLA는 어떻게 정의하셨나요?

✅ **레이어 기준:**
- **Bronze(Raw)**: 원천 그대로 적재. 스키마 변환 없음, 파티션은 수집일(ingested_at) 기준. 소스 데이터의 단순 미러로 재처리 시 원점으로 돌아갈 수 있는 보험 역할입니다.
- **Silver(DW)**: 중복 제거, 타입 캐스팅, 비즈니스 키 표준화, 도메인 간 조인 적용. 데이터를 '쓸 수 있는 상태'로 만드는 레이어입니다.
- **Gold(Mart)**: 비즈니스 로직이 반영된 집계 테이블, KPI 마트. 실무 조직이 직접 사용하는 레이어입니다.

**SLA 정의:**
- Bronze: 원천 이벤트 발생 후 30분 이내 적재
- Silver: Bronze 적재 완료 후 1시간 이내
- Gold: 배치는 매일 오전 9시(영업 시작 전) 갱신 완료, 실시간 집계는 ClickHouse MV 기준 5분 이내

**꼬리질문**

- 4-2. Silver Layer에서 데이터 품질 체크를 실패했을 때 Gold Layer로 흘러가지 못하도록 어떻게 막으셨나요?

🟢 Spark 처리 단에서 스키마 및 비즈니스 로직 검증을 코드 레벨로 강제하고 있습니다. 검증 실패 시 task를 fail시켜 Airflow 의존성으로 인해 Gold 레이어 DAG가 자동으로 실행되지 않도록 막습니다. 스트리밍 파이프라인처럼 가용성이 품질보다 우선되는 경우에는 검증 실패 레코드를 DLQ(Dead Letter Queue)로 분리하고 Slack 알림으로 감지하는 방식으로 무중단성과 품질을 동시에 확보하고 있습니다.

- 4-3. 190개 테이블을 혼자 관리하면서 레이어별 테이블 네이밍/스키마 표준을 어떻게 일관성 있게 유지하셨나요?

🟢 `datalake_`, `warehouse_`, `datamart_` prefix로 레이어를 구분하고, 그 뒤에 `payment_`, `user_`, `ad_` 같은 도메인 prefix를 붙여 도메인 단위로 테이블을 분리하고 있습니다. 스키마 정의는 코드로 관리하여 임의 변경을 방지했고, 새 테이블 추가 시에는 네이밍 규칙을 따르도록 내부 문서화했습니다.

## PART 2. Apache Iceberg 실무 검증 (PT + 전화면접 심화)

### Q5. Iceberg의 hidden partitioning을 사용하셨다고 하셨는데, 셔클 서비스 맥락에서 차량 운행 데이터를 저장한다면 파티션 전략을 어떻게 설계하시겠습니까?

🟢 차량 운행 데이터는 시계열 이벤트이므로 이벤트 발생 시간(event_at)을 기준으로 시간 단위 파티셔닝을 우선 적용하겠습니다. 분석 패턴에서 지역별 조회가 많다면 서비스 지역(region_id)이나 운행 단위(trip_id의 상위 그룹)를 복합 파티셔닝 키로 추가하는 것을 고려하겠습니다. Iceberg hidden partitioning의 장점을 살려 파티션 컬럼을 쿼리에 명시하지 않아도 프루닝이 자동으로 되도록 설계하겠습니다.

**꼬리질문**

- 5-1. 차량 ID + 날짜로 파티셔닝했을 때, 특정 차량의 전체 경로를 조회하는 쿼리와 특정 지역의 모든 차량을 조회하는 쿼리 성능이 극단적으로 다를 것입니다. 어떻게 이 상충관계를 해소하시겠습니까?

🔧 **[기존 답변 수정]** "프로젝션" 언급은 ClickHouse의 개념으로, Iceberg 맥락에서는 적절하지 않습니다. Iceberg에서의 실제 해결 방법을 사용해야 합니다.

> 두 쿼리 패턴의 특성이 완전히 다르기 때문에 단일 파티션 전략으로 양쪽 모두를 최적화하기는 어렵습니다. 실용적인 접근으로는 파티션을 date 기준으로 잡고(시간 범위 필터가 대부분의 쿼리에 포함되므로), 특정 차량 경로 조회는 Iceberg의 파일 레벨 통계(min/max on vehicle_id)를 활용해 불필요한 파일을 pruning하도록 합니다. 지역별 조회처럼 접근 패턴이 명확히 다른 경우에는 Gold 레이어에서 지역 기준으로 pre-aggregated된 별도 테이블로 서빙하는 방식이 더 현실적입니다.

- 5-2. Iceberg partition evolution을 실제로 수행해보셨나요? 운영 중인 테이블 파티션 전략을 바꿀 때 어떤 리스크가 있고 어떻게 대응하셨나요?

🟢 파티션 스펙을 변경하면 신규 데이터는 새 파티션 구조로 적재되고 기존 데이터는 구 구조를 유지합니다. 이 경우 두 파티션 구조가 혼재하는 기간이 생겨, 일부 쿼리에서 전체 스캔이 발생할 수 있습니다. 실제 대응으로는 파티션 변경 후 히스토리 데이터를 새 파티션 구조로 rewrite하는 작업을 별도로 진행했고, 다운스트림 파이프라인이 파티션 컬럼을 하드코딩하고 있는 경우가 없는지 사전 파악을 먼저 했습니다.

- 5-3. Iceberg v2 row-level delete(Delete File)를 사용해보셨나요? Copy-on-Write vs Merge-on-Read 모드를 실제로 어떻게 선택하셨나요?

🟢 현재는 대부분 배치성 테이블이고 BI 조회 위주의 사용 패턴이라 COW를 기본으로 사용하고 있습니다. COW는 write 시에 전체 파일을 새로 씁니다. 읽기 성능이 우선인 환경에서 적합합니다. 반면 MOR은 Delete File을 별도로 기록해 write 속도는 빠르지만, 읽기 시 병합 오버헤드가 생기고 compaction 전략을 더 촘촘하게 운영해야 합니다. 향후 고빈도 UPSERT가 필요한 CDC 기반 테이블이 늘어나면 해당 테이블만 MOR로 전환하는 것을 고려하고 있습니다.

### Q6. "일 1억건 이상의 데이터를 통합"하셨다고 PT에 나와있는데, Iceberg에서 이 규모의 데이터 compaction을 어떻게 운영하셨나요?

🟢 대부분의 테이블이 배치성 적재이고 실시간 파이프라인도 append 기반 증분 적재라, small file이 급격히 누적되는 구조는 아닙니다. 현재는 전체 테이블 대상으로 7일에 한 번 compaction 및 snapshot 정리를 Airflow DAG로 실행하고 있습니다. 테이블별 적재 주기와 업데이트 빈도가 다양해지면, 빈도가 높은 테이블은 별도로 더 짧은 compaction 주기를 적용하는 방식으로 전략을 분리할 것입니다.

**꼬리질문**

- 6-1. `rewriteDataFiles`를 Airflow로 주기적으로 돌리셨다고 하셨는데, compaction 작업이 운영 쿼리와 리소스를 경합하는 문제는 없었나요? 어떻게 격리하셨나요?

🟢 배치 적재가 가장 적은 시간대(새벽)에 compaction을 실행하고 있습니다. OCC 충돌 리스크를 최소화하기 위해 compaction 대상을 현재 시점 기준 7일 이전 파티션으로 제한합니다. 활발하게 write가 일어나는 최근 파티션은 건드리지 않는 전략입니다.

- 6-2. Compaction 도중 다른 write job이 동시에 실행되면 Iceberg OCC 충돌이 발생할 수 있습니다. 실제로 이런 상황을 경험하셨나요?

🟢 7일 이전 파티션만 대상으로 하기 때문에 동시 write와의 충돌은 거의 발생하지 않았습니다. 다만 간혹 발생할 수 있는 OCC 충돌에 대비해 Airflow task에 retry 설정과 timeout을 걸어두어, 충돌이 발생해도 재시도로 자동 복구되도록 장치했습니다.

- 6-3. Snapshot이 쌓이면 plan overhead가 증가합니다. `expireSnapshots`를 언제, 얼마나 자주 실행하셨나요? 그 기준은?

🟢 일주일에 한 번, 7일 이전 스냅샷 중 매일의 자정 기준 마지막 스냅샷만 남기고 나머지를 정리합니다. 운영에서 KYC 인증, 고객 리스크 파악 등의 로직이 일 단위 Time Travel을 사용하고 있기 때문에 일별 기준점은 반드시 보존해야 합니다. 이 방식으로 최대 연 365개 스냅샷이 유지되는데, Iceberg 가이드 기준 1000~2000개 이하는 성능상 용인 가능한 수준이라 현재는 개수 제한 없이 운영 중입니다. 다만 5년 이상 누적 시 성능 병목 가능성이 있어 향후 정책 보완을 고려하고 있습니다.

## PART 3. Spark 퍼포먼스 & 실시간 처리

### Q7. Spark Structured Streaming을 실제 운영에서 사용하셨는데, 마이크로배치 간격을 어떻게 설정하셨고, trigger 방식은 무엇을 선택하셨나요?

🟢 비즈니스 요구사항에 따라 다르게 적용했습니다. 광고 서빙처럼 실시간 반영이 요구되는 경우는 `processingTime='30 seconds'`, 앱 로그처럼 BI에 활용되는 데이터는 `processingTime='3 minutes'`로 설정했습니다. 리소스를 상시 유지해야 하는 부담이 있어 비용 최적화가 필요한 케이스는 `availableNow=True` trigger를 활용해 처리 완료 후 클러스터를 종료하는 방식도 검토했습니다.

**꼬리질문**

- 7-1. `trigger(processingTime='30 seconds')`와 `trigger(availableNow=True)` 방식의 차이를 설명하고, 각각 어떤 상황에 적합한지 말씀해 주세요.

🟢 `processingTime`은 Spark Executor가 항상 올라와 있는 상태로 지정 주기마다 마이크로 배치가 트리거됩니다. 연속적인 실시간 처리에 적합합니다. `availableNow`는 배치처럼 트리거 시점에 현재까지 쌓인 모든 데이터를 처리하고 스스로 종료합니다. 배치와 다른 점은 Streaming context의 체크포인트를 그대로 활용하기 때문에 offset 기반 정합성 관리가 가능하다는 것입니다. 리소스 상시 유지 비용을 줄이고 싶을 때, 또는 Airflow 스케줄로 트리거하고 싶을 때 `availableNow`가 유리합니다.

- 7-2. Kafka 토픽의 파티션 수와 Spark Streaming의 파티션 수가 맞지 않을 때 어떤 문제가 생기고 어떻게 조정하셨나요?

🟢 Spark Structured Streaming은 Kafka 파티션 수 = Spark 파티션 수로 1:1 매핑됩니다. Kafka 파티션이 적으면 일부 Executor 코어가 idle 상태로 리소스만 낭비합니다. 반대로 처리 후 무거운 연산이 있다면 Kafka에서 읽어온 후 `repartition()`으로 병렬도를 높여 처리 성능을 개선할 수 있습니다. 실제로 Kafka 파티션 수와 Executor 코어 수를 맞추는 것을 기준으로 설정했습니다.

- 7-3. Streaming 작업이 Lag을 못 따라잡는 상황(consumer lag 누적)이 발생했을 때 어떻게 대응하셨나요?

🔧 **[기존 답변 보완]** 7-2를 그대로 참조하는 답변은 면접에서 빈약하게 보입니다. 원인 진단부터 단계적으로 설명해야 합니다.

> 먼저 lag 원인을 두 가지로 구분합니다. 처리 로직 자체가 느린 경우(연산 병목)와, 데이터가 갑자기 폭증한 경우입니다. 처리 로직 병목이라면 `repartition()`으로 병렬도를 높이거나 Spark UI에서 skew 여부를 확인합니다. 데이터 폭증이라면 `maxOffsetsPerTrigger`를 높여 배치당 처리량을 늘리거나, 카프카 파티션 수를 증설해 소비 병렬도를 올립니다. 단기적으로는 기존 배치가 처리를 따라잡을 때까지 트리거 간격을 줄이고 리소스를 임시 증설했습니다.

### Q8. PT에서 "광고 트래픽 10배 증가"라는 성과를 언급하셨는데, 이게 데이터 팀의 기여로 볼 수 있는 근거가 뭔가요?

🟢 광고 지면 확대와 사업팀 영업 노력이 복합적으로 작용했다는 것을 인정합니다. 다만 데이터 플랫폼 기여를 말씀드리면, 유저 세그먼트 기반 타겟팅 파이프라인 구축 이전에는 광고주에게 제공 가능한 정량 지표가 없어 사업 확장에 한계가 있었습니다. 실시간 CTR 집계와 유저 피처 서빙 인프라가 갖춰지면서 광고주에게 신뢰할 수 있는 성과 지표를 제공하고, 타겟팅 정확도를 개선하는 피드백 루프가 가능해진 것이 성장의 기반이 됐다고 봅니다.

**꼬리질문**

- 8-1. 광고 트래픽 증가가 데이터 플랫폼 구축 덕분인지, 광고 사업팀의 영업 확대 덕분인지 어떻게 구분할 수 있나요?

🟢 단일 원인으로 귀속시키는 것은 어렵고 솔직히 그렇게 주장하지도 않겠습니다. 다만 데이터 플랫폼이 가능하게 한 부분은 명확합니다. CTR·CVR 같은 지표 없이는 광고주 영업 자체가 어렵고, 유저 세그먼트 없는 노출은 전환율이 낮습니다. 플랫폼 구축 이후 이 두 가지가 가능해졌기 때문에 트래픽 증가의 필요 조건을 데이터 팀이 제공했다고 말씀드릴 수 있습니다.

- 8-2. 그 기간 동안 Retention, CTR 등 광고 효율 지표는 어떻게 변했나요? 트래픽 볼륨 외에 품질 개선을 나타내는 지표가 있나요?

🟢 유저 세그먼트 타겟팅 적용 이후 CTR은 지속적으로 개선 추세입니다. 다만 광고 플랫폼이 아직 초기 단계라 A/B 테스트 인프라가 갖춰지지 않아 정확한 귀속 분석은 어려운 상황입니다. Retention의 경우 여행 도메인 특성상 방문 주기 자체가 길어 단기 지표로 유의미한 변화를 확인하기는 어렵습니다. 이 부분은 현재의 명확한 한계로 인식하고 있으며, 향후 실험 플랫폼 구축이 필요한 영역입니다.

## PART 4. 데이터 거버넌스 & 품질 (셔클 JD 핵심 요구사항)

### Q10. 셔클 데이터팀 JD에 "데이터 리니지 시스템 고도화"가 명시되어 있습니다. 지금까지 경험에서 Airflow DAG 의존성으로 리니지를 대체하셨다고 하셨는데, 컬럼 레벨 리니지가 없어서 실제로 문제가 된 경험이 있었나요?

🟢 1인 조직이었기 때문에 대부분의 테이블과 컬럼 단위 흐름을 제가 직접 알고 있어 큰 문제는 없었습니다. 다만 사용자 입장에서 시각적으로 확인할 수 있는 수단이 없어, 데이터 출처나 변환 로직에 대한 문의가 결국 저에게 집중되는 병목이 생겼습니다. 또한 신규 파이프라인 추가 시 영향도를 파악하는 데 수작업으로 DAG를 추적해야 하는 비효율이 있었습니다. 이 경험이 DataHub 도입을 향후 과제로 설정한 배경입니다.

**꼬리질문**

- 10-1. DataHub를 직접 운영한 경험이 없는데, 셔클에서 DataHub 도입을 맡게 된다면 어떤 순서로 어떻게 접근하시겠나요?

✅ 도입 순서는 다음과 같이 접근하겠습니다.

> 1단계: 현황 파악 — 기존 Glue Catalog, Airflow DAG, S3 버킷을 DataHub의 메타데이터 크롤러로 연결해 자산 목록을 자동 수집합니다. 이미 존재하는 구조물을 인벤토리화하는 것이 첫 번째 가치입니다.  
> 2단계: 오너십 정의 — 도메인별 테이블 오너십(owner 태그)을 정의하고 DataHub에 등록합니다. 카탈로그가 비어 있으면 아무도 사용하지 않습니다.  
> 3단계: 리니지 연동 — Spark 파이프라인에 OpenLineage emitter를 추가해 컬럼 레벨 리니지를 자동 수집합니다. Airflow에도 DataHub 오퍼레이터를 연결합니다.  
> 4단계: 데이터 품질 연동 — 기존 품질 체크 결과를 DataHub의 Dataset Health로 연동해 카탈로그 내에서 품질 상태도 확인 가능하게 만듭니다.  
> 처음부터 완벽하게 구축하려 하기보다 빠른 가치 제공(자산 목록 → 오너십 → 리니지 순)으로 팀 내 신뢰를 쌓는 방식으로 접근하겠습니다.

- 10-2. OpenLineage, Marquez 같은 오픈소스 리니지 도구와 DataHub의 차이를 알고 계신가요? 셔클 환경에 어떤 것이 더 적합할까요?

✅ OpenLineage는 리니지 이벤트를 생성하는 표준 스펙이고, Marquez는 그 스펙의 경량 구현체입니다. 리니지 추적 자체에 집중되어 있고 운영이 단순합니다. DataHub는 리니지 외에도 메타데이터 관리, 데이터 사전, 오너십, 품질 지표, 검색 기능을 포함하는 종합 데이터 카탈로그 플랫폼입니다.

> 셔클 JD에 "데이터 표준화 정책, 데이터 사전 고도화, 리니지 시스템"이 모두 언급되어 있는 것으로 보면, 단순 리니지 추적만이 아니라 전사 데이터 거버넌스 체계가 목표인 것으로 보입니다. 이 경우 DataHub가 더 적합합니다. 다만 DataHub는 운영 복잡도가 높기 때문에, 초기에는 Managed DataHub(Acryl Data)나 기존 AWS Glue Catalog를 DataHub에 연동하는 방식으로 운영 부담을 줄이는 것을 권장합니다.

### Q11. 데이터 품질 모니터링에서 "row count 급감, null rate 임계치 초과"를 체크하셨다고 했는데, 셔클의 차량 운행 데이터에서는 어떤 품질 지표가 중요할까요?

✅ 차량 운행 데이터 특성상 다음 품질 지표가 중요하다고 생각합니다.

> - **이벤트 신선도(freshness)**: 차량별 마지막 이벤트 수신 시간. 일정 시간 이상 신호가 없으면 파이프라인 장애 또는 차량 이슈 가능성.
> - **GPS 유효성**: 위경도 값이 서비스 운행 지역 범위 내에 있는지, 연속된 두 좌표 간 이동 속도가 물리적으로 가능한 범위인지 (예: 300km/h 초과 = 노이즈).
> - **이벤트 시퀀스 완결성**: 호출 → 배차 → 승차 → 하차의 이벤트 체인이 정상적으로 완결되는지, 중간 이벤트가 누락된 trip_id 비율.
> - **중복 이벤트 비율**: 동일 event_id가 복수로 들어오는 경우 (네트워크 재전송 등 원인).
> - **배차 성공률 데이터 정합성**: 호출 건수 대비 배차 완료 건수의 비율이 합리적인 범위인지 (급격한 이상값은 파이프라인 문제 가능성).

**꼬리질문**

- 11-1. 예를 들어 특정 지역의 차량 GPS 신호가 30분간 수신되지 않을 때, 이게 데이터 파이프라인 장애인지 실제 차량 운행 이슈인지 어떻게 구분하시겠나요?

> - **운영팀 교차 확인**: 데이터 지표 이상 탐지 시 운영팀의 차량 상태 대시보드와 비교해 이중 검증합니다.

- 11-2. Great Expectations, dbt test, Soda 같은 데이터 품질 프레임워크를 사용해보셨나요? 사용하지 않으셨다면 직접 구현한 방식과 비교해서 어떤 차이가 있을까요?

🟢 별도 품질 프레임워크는 사용하지 않았습니다. 1인 조직 특성상 생산성과 운영 효율을 우선해, 비즈니스 로직 검증은 원천 도메인에 위임하고 데이터 레이크에는 원천 그대로를 적재하는 것이 정합성 측면에서 맞다고 판단했습니다. Gold 레이어의 서빙 데이터에 한해서만 Spark 코드 레벨로 스키마와 핵심 규칙을 강제하는 방식을 사용했습니다. 직접 구현 대비 Great Expectations나 Soda의 장점은 테스트 정의를 코드가 아닌 YAML로 선언해 비개발자도 규칙을 확인·추가할 수 있고, 품질 결과를 대시보드로 시각화할 수 있다는 점입니다. 셔클처럼 팀 규모가 커지고 거버넌스 요구사항이 높아지는 환경에서는 도입 가치가 충분하다고 생각합니다.

🔧 **[기존 답변 오타 수정 및 보완]** "적합하도 생각합니다" → "적합하다고 생각합니다"

## PART 6. 셔클 도메인 적합성 & 모빌리티 데이터

### Q14. PT 지원 동기에서 "IoT 데이터를 효율적으로 처리하는 안정적인 인프라"를 언급하셨는데, 차량 IoT 데이터를 직접 다뤄본 경험이 없습니다. 이 영역에서 어떻게 빠르게 기여하시겠습니까?

✅ 솔직하게 말씀드리면 차량 IoT 데이터를 직접 다뤄본 경험은 없습니다. 다만 데이터 파이프라인의 핵심 구조인 수집(Kafka) → 처리(Spark/Flink) → 저장(Iceberg) → 서빙(ClickHouse) 아키텍처는 도메인에 관계없이 동일합니다. 실제로 핀테크에서 초당 수만 건의 결제·광고 이벤트를 처리한 경험이 있어, 고처리량 이벤트 파이프라인 설계와 운영에는 즉시 기여할 수 있습니다. 차량 도메인 특유의 개념(GPS 보정, OD 분석, 지오펜싱 등)은 빠르게 학습하겠으며, 초기에는 기존 팀원들이 잘 아는 도메인 지식을 존중하고 인프라 안정성과 파이프라인 신뢰성 측면에서 우선 기여하겠습니다.

**꼬리질문**

- 14-1. 차량 GPS 데이터는 초당 수십 건, 차량 수백 대 규모면 초당 수만 건의 이벤트입니다. 기존에 다뤄보신 핀테크 이벤트 데이터(광고 클릭/노출)와 처리 패턴이 어떻게 다르다고 생각하시나요?

✅ 볼륨 특성은 비슷하지만 처리 패턴에서 중요한 차이가 있습니다.

> - **연속성과 순서**: GPS 이벤트는 동일 차량에서 연속으로 발생하며, 이벤트 순서가 경로 재구성에 결정적입니다. 광고 클릭은 상호 독립적 이벤트입니다. 따라서 GPS 처리에서는 이벤트 시간 기반 ordering과 device_id별 상태 관리가 중요합니다.
> - **공간 정보**: 위경도 데이터는 H3·Geohash 같은 공간 인덱스가 필요합니다. 광고 데이터에는 없는 차원입니다.
> - **노이즈 비율**: GPS는 하드웨어 특성상 노이즈(튀는 좌표)가 광고 이벤트보다 훨씬 높습니다. 필터링과 보정 로직이 파이프라인에 포함되어야 합니다.
> - **상태 기반 처리**: 운행 중/운행 완료/대기 같은 차량 상태를 파이프라인 내에서 추적해야 합니다. Flink의 Keyed State가 적합한 영역입니다.

- 14-2. 시계열 데이터에 특화된 TimescaleDB, InfluxDB, Apache Druid 같은 솔루션이 있는데, ClickHouse 대신 이런 도구를 셔클에 도입하는 것은 어떻게 생각하시나요?

✅ 각 도구의 특성을 비교해서 말씀드리겠습니다.

> - **InfluxDB**: IoT 메트릭에 최적화되어 있지만 JOIN·복잡한 집계가 약하고 SQL이 아닌 Flux 쿼리 언어를 배워야 합니다. 차량 운행 데이터처럼 이벤트와 디멘션 조인이 필요한 분석에는 적합하지 않습니다.
> - **TimescaleDB**: PostgreSQL 기반이라 SQL 친화적이지만 수억 건 규모에서 ClickHouse 대비 집계 성능이 떨어집니다.
> - **Druid**: 고카디널리티 실시간 분석에 강하지만 운영 복잡도가 높고 학습 비용이 큽니다.
> - **ClickHouse**: 이미 팀이 익숙한 환경이고, 시계열 + 관계형 혼합 쿼리를 잘 처리하며 Materialized View로 집계를 사전 계산할 수 있습니다. 차량 데이터 규모에서도 충분한 성능을 보여줍니다.
> 
> 현시점에서는 운영 안정성과 학습 비용을 고려해 ClickHouse를 유지하는 것이 맞습니다. 단, 수십만 대 규모로 성장해 고카디널리티 실시간 집계가 병목이 된다면 Druid 전환을 검토할 것입니다.

- 14-3. 셔클의 AI 다이내믹 라우팅이 실시간으로 동작하려면 차량 위치 데이터가 수 초 이내에 서빙되어야 할 것입니다. 데이터 레이턴시 요구사항이 이렇게 엄격할 때 현재 아키텍처(Spark Streaming 마이크로배치)로 대응 가능하다고 보시나요?

✅ AI 다이내믹 라우팅처럼 밀리초~수 초 레이턴시가 요구되는 실시간 의사결정은 Spark Streaming 마이크로배치로는 한계가 있다고 봅니다. 다만 이 경우 데이터 엔지니어링 레이어의 역할을 명확히 구분해야 합니다.

> - **라우팅 의사결정 레이어** (수 초 이내): 차량 현재 위치는 Kafka Consumer → Redis/DynamoDB 같은 인메모리 저장소로 직접 서빙하는 구조가 필요합니다. Spark를 거치지 않는 별도 경로입니다.
> - **데이터 플랫폼 레이어** (분석·모니터링): Spark Streaming이 담당하는 운행 통계, KPI 집계, 이력 관리는 수 초~수 분 레이턴시가 허용 가능합니다.
> 
> 즉 Spark Streaming은 분석 파이프라인에 적합하고, 실시간 라우팅 의사결정에 필요한 현재 상태 서빙은 별도 스트림 처리(Flink 또는 Kafka Streams)와 인메모리 스토어 조합이 필요합니다.

### Q15. 셔클은 승객 호출 → AI 경로 최적화 → 차량 배차 → 운행 완료의 사이클에서 데이터가 발생합니다. 이 비즈니스 플로우에서 데이터 엔지니어로서 가장 먼저 어떤 파이프라인을 구축해야 한다고 생각하시나요?

✅ 가장 먼저 구축해야 할 것은 **Trip Lifecycle 이벤트 파이프라인**입니다.

> 호출→배차→승차→하차의 이벤트 체인이 데이터 플랫폼의 원천 사실(Source of Truth)이 됩니다. 이 파이프라인 없이는 배차 성공률, 평균 대기시간, 경로 효율성 같은 KPI를 하나도 계산할 수 없습니다. 기술적으로는 Kafka에서 trip 이벤트를 수집 → Spark Streaming으로 이벤트 체인 완결 여부를 상태 기반으로 추적 → Iceberg에 trip 단위로 적재하는 구조입니다. 두 번째 우선순위는 차량 위치 파이프라인(GPS 데이터), 세 번째가 KPI 집계 파이프라인입니다.

**꼬리질문**

- 15-1. 배차 성공률, 평균 대기시간, 경로 효율성 같은 핵심 KPI를 실시간으로 모니터링하려면 어떤 데이터 파이프라인이 필요할까요?

✅ 각 KPI별로 필요한 파이프라인을 말씀드리겠습니다.

> - **배차 성공률**: Kafka에서 호출 이벤트와 배차 완료 이벤트를 5분 윈도우로 집계 → ClickHouse Materialized View로 `dispatched / requested` 비율 실시간 계산.
> - **평균 대기시간**: 호출 timestamp와 배차 timestamp의 차이를 이벤트별로 계산 → ClickHouse에 trip_id별 wait_time 컬럼 저장 → MV로 이동 평균 집계.
> - **경로 효율성**: 실제 운행 거리(GPS trajectory) vs 최적 경로 거리(라우팅 엔진 출력) 비교 → 배치로 daily 집계가 현실적. 실시간 계산은 GPS 데이터 처리 비용이 크기 때문에 배치로 선처리 후 ClickHouse에 적재.
> 
> 공통 아키텍처: Kafka → Spark Streaming → Iceberg(원천) → ClickHouse MV(서빙) → Superset 대시보드.

- 15-2. 셔클 서비스가 헝가리 등 해외로 확장 중입니다. 다국가 데이터를 통합 처리할 때 어떤 추가적인 설계 고려사항이 필요할까요?

✅ 크게 세 가지 영역에서 고려가 필요합니다.

> - **타임존**: 모든 timestamp는 UTC로 저장하고 표시 레이어에서 지역 타임존 변환. 소스 시스템이 로컬 타임존으로 timestamp를 기록하는 경우 수집 시점에 UTC 정규화가 필수입니다.
> - **데이터 주권(Data Sovereignty)**: EU의 GDPR은 개인 데이터의 EU 역외 이전을 제한합니다. 헝가리 사용자의 위치·탑승 이력 데이터는 EU 리전에 저장하거나, 비식별화 후 한국으로 전송하는 설계가 필요합니다.
> - **멀티테넌시**: 모든 테이블에 `country_code` 파티션 키를 추가해 국가별 데이터를 논리적으로 분리하고, 집계 시에도 국가 단위 drill-down이 가능한 구조로 설계합니다.

## PART 7. 장애 대응 & 운영 실무

### Q16. 운영 중 가장 심각했던 데이터 파이프라인 장애를 하나 말씀해주세요. 무엇이 원인이었고, 어떻게 탐지했으며, 복구까지 얼마나 걸렸나요?

✅ **[실제 경험 기반으로 반드시 본인 사례로 교체하세요. 아래는 구조적 가이드 + 예시 시나리오입니다]**

> ClickHouse 도입 초기, Buffer 엔진을 적용하기 전에 Spark Streaming이 30초 간격으로 소량씩 insert를 반복하면서 "too many parts" 에러가 발생해 ClickHouse 클러스터가 write를 거부한 적이 있었습니다. Airflow 알림으로 Streaming job 실패를 탐지했고, ClickHouse 로그를 확인해 parts 누적이 원인임을 파악했습니다. 즉각 대응으로는 `OPTIMIZE TABLE FINAL` 명령으로 수동 병합을 실행해 클러스터를 정상화했습니다. 복구까지 약 40분이 걸렸고, 그 사이 ClickHouse 집계 대시보드에 데이터 공백이 발생했습니다. 재발 방지로 Buffer 엔진을 도입하고, too many parts 임계치에 대한 모니터링 알림을 추가했습니다.

**꼬리질문**

- 16-1. 그 장애가 다운스트림 BI 대시보드나 비즈니스 팀에 영향을 미쳤나요? 비개발 직군에게 어떻게 커뮤니케이션 하셨나요?

✅ 광고 성과 대시보드에 40분간 데이터 공백이 발생했습니다. 비개발 직군에게는 기술적 원인을 상세히 설명하기보다, "현재 실시간 데이터 수집에 문제가 발생했고 복구 작업 중입니다. 예상 복구 시간은 OO시입니다"와 같이 영향 범위와 예상 복구 시간 중심으로 간결하게 공유했습니다. 복구 완료 후에는 공백 기간 데이터가 Iceberg에 보존되어 있어 재적재로 대시보드를 정상화했다는 후속 통보를 했습니다.

- 16-2. 같은 장애가 재발하지 않도록 어떤 사후 조치를 하셨나요?

✅ 세 가지 조치를 했습니다.

> 1. **근본 원인 해소**: Buffer 엔진 도입으로 insert 빈도를 줄여 구조적으로 too many parts 발생을 방지.
> 2. **조기 감지 강화**: ClickHouse의 `system.parts` 테이블을 주기적으로 조회해 파트 수가 임계치(예: 300)에 근접하면 Slack 알림을 발송하는 모니터링 추가.
> 3. **복구 자동화**: 수동 `OPTIMIZE` 명령을 스크립트화해 장애 발생 시 즉시 실행 가능하도록 준비.

- 16-3. 장애 탐지부터 복구까지의 MTTR(Mean Time To Recovery)을 어떻게 줄이기 위해 노력하셨나요?

✅ MTTR을 줄이는 데 가장 효과적인 것은 런북(Runbook)과 모니터링 체계입니다.

> - **탐지 속도 개선**: 장애 유형별 알림 채널과 심각도를 구분해, P1 장애(서비스 영향)는 즉시 Slack 알림, P2는 이메일로 분류해 불필요한 노이즈를 줄였습니다.
> - **진단 속도 개선**: 장애 유형별 체크리스트를 런북으로 만들어, 처음 보는 증상도 빠르게 원인을 좁힐 수 있게 했습니다.
> - **복구 속도 개선**: 재처리·수동 복구 스크립트를 미리 작성해두어, 장애 시 명령어를 새로 짜지 않아도 즉시 실행 가능하도록 준비했습니다.

### Q17. Airflow DAG 수가 늘어날수록 스케줄러 성능 이슈가 발생할 수 있습니다. 실제로 이런 문제를 경험하셨나요? 운영 규모와 어떻게 대응하셨나요?

✅ DAG 수가 50개를 넘어가면서 스케줄러 heartbeat 지연과 DAG 파싱 시간이 증가하는 현상을 경험했습니다.

> 대응으로는 첫째, DAG 파일당 로직을 최소화하고 공통 유틸리티를 별도 모듈로 분리해 파싱 시간을 줄였습니다. 둘째, `dagbag_import_timeout` 설정을 조정하고 DAG Serialization을 활성화했습니다. 셋째, 불필요하게 짧은 poke_interval을 가진 Sensor를 점검해 스케줄러 부하를 줄였습니다. 현재 규모에서는 이 조치로 안정적으로 운영 중입니다.

**꼬리질문**

- 17-1. Airflow를 EMR이나 EKS 위에서 운영하셨나요? 인프라 구성을 어떻게 하셨나요?

✅ AWS 환경에서 MWAA(Amazon Managed Workflows for Apache Airflow)를 사용했습니다. 자체 클러스터 운영 대신 관리형 서비스를 선택한 이유는 1인 운영 환경에서 Airflow 인프라 자체의 유지보수 비용을 줄이기 위해서였습니다. 실제 Spark 작업은 EMR on EKS 또는 EMR Serverless로 submit하는 방식으로, MWAA가 오케스트레이터 역할만 하고 컴퓨팅은 분리된 구조를 사용했습니다.

- 17-2. DAG 의존성이 복잡해질 때 (Cross-DAG dependency, external sensor) 어떻게 관리하셨나요?

✅ Cross-DAG 의존성은 `ExternalTaskSensor`를 사용해 다른 DAG의 특정 task 완료를 기다리는 방식으로 구현했습니다. 다만 ExternalTaskSensor는 poke 방식이라 스케줄러 부하를 줄이기 위해 `mode='reschedule'`을 사용했습니다. 의존성 관계가 복잡해질수록 DAG 구조 자체가 리니지 역할을 해야 해서, 의존 방향을 명확히 문서화하고 순환 의존이 생기지 않도록 주의했습니다.

- 17-3. Airflow 대신 Prefect, Dagster, dbt + Airflow 조합 등을 검토해보셨나요? 셔클 환경에 어떤 오케스트레이션 도구가 적합하다고 생각하시나요?

✅ Dagster는 asset 단위의 의존성 관리와 내장된 데이터 카탈로그 기능이 매력적이지만, 학습 곡선이 있고 기존 Airflow 자산을 마이그레이션하는 비용이 큽니다. Prefect는 Python native하고 UI가 직관적이지만 엔터프라이즈 기능은 유료입니다. 셔클처럼 AWS 기반 환경에서 이미 MWAA를 사용 중이라면, 추가 마이그레이션 비용 없이 Airflow를 유지하면서 dbt를 데이터 변환 레이어로 도입해 SQL 변환 로직을 체계화하는 방향이 현실적이라고 생각합니다.

## PART 8. 기술 깊이 검증 (즉답 요구)

### Q18. 즉답 압박 — 아래 질문에 간결하게 답해주세요.

- 18-1. Kafka에서 `auto.offset.reset=earliest`와 `latest`의 차이는? 실제로 어떤 걸 사용하셨나요?

✅ `earliest`: consumer group에 커밋된 offset이 없을 때 토픽의 가장 오래된 메시지부터 소비합니다. `latest`: 커밋된 offset이 없을 때 컨슈머 시작 시점 이후의 새 메시지만 소비합니다. 데이터 파이프라인에서는 메시지를 빠뜨리면 안 되기 때문에 `earliest`를 사용했습니다. `latest`는 이미 처리된 히스토리 데이터가 필요 없고 새 이벤트만 실시간 모니터링하는 경우에 적합합니다.

- 18-2. Spark에서 `repartition()`과 `coalesce()`의 차이는? 언제 어떤 걸 쓰나요?

✅ `repartition(n)`: 전체 셔플이 발생하며 파티션 수를 늘리거나 줄일 수 있고, 균등한 데이터 분포를 만듭니다. `coalesce(n)`: 셔플 없이 기존 파티션을 합치는 방식으로 파티션 수를 줄일 때만 사용 가능합니다. 대용량 write 전에 파일 수를 줄이고 싶을 때는 `coalesce`(셔플 없이 비용이 저렴), 데이터 skew를 해소하거나 파티션 수를 늘릴 때는 `repartition`을 사용합니다.

- 18-3. Iceberg의 `overwrite` 모드와 `append` 모드의 차이는? 멱등성 측면에서 어떤 게 안전한가요?

✅ `overwrite`: 지정 파티션의 기존 데이터를 교체합니다. `append`: 기존 데이터에 새 레코드를 추가합니다. 멱등성 측면에서는 `overwrite`가 안전합니다. `append`는 재실행 시 동일 데이터가 중복으로 쌓이는 반면, 파티션 단위 `dynamic overwrite`는 재실행해도 동일 결과를 보장합니다. 배치 파이프라인에서는 대부분 `overwrite`를 기본으로 사용합니다.

- 18-4. ClickHouse의 `ReplicatedMergeTree`와 `MergeTree`의 차이는?

✅ `MergeTree`: 단일 노드용 테이블 엔진. 복제 없이 로컬에만 데이터를 저장합니다. `ReplicatedMergeTree`: ZooKeeper 또는 ClickHouse Keeper를 통해 노드 간 데이터를 복제합니다. 고가용성(HA) 구성이 필요한 프로덕션 환경에서는 `ReplicatedMergeTree`를 사용해야 노드 장애 시에도 데이터 유실 없이 서비스를 지속할 수 있습니다.

- 18-5. Spark `broadcast join`의 기본 threshold는 얼마이고, 어떤 상황에서 명시적으로 힌트를 주시나요?

✅ 기본값은 `spark.sql.autoBroadcastJoinThreshold = 10MB`입니다. AQE가 통계를 정확히 수집하지 못해 자동으로 broadcast를 선택하지 않을 때, 또는 테이블이 10MB를 조금 초과하지만 실제로 broadcast가 충분히 가능한 크기(예: 50~100MB)일 때 `broadcast()` 힌트를 명시적으로 사용합니다. 특히 실행 계획에서 SortMergeJoin이 보이는데 한쪽 테이블이 작은 lookup 테이블임을 알고 있는 경우에 유용합니다.

- 18-6. S3에서 작은 파일이 많을 때 발생하는 구체적인 성능 문제는 무엇이고 어떻게 해결하나요?

✅ S3는 파일 하나당 HTTP 요청이 발생합니다. 작은 파일이 수만 개 있으면 LIST + GET 요청 수가 폭증해 레이턴시가 증가하고 S3 요청 비용이 올라갑니다. Spark 쪽에서는 작은 파일을 열고 닫는 오버헤드가 증가해 처리 성능이 저하됩니다. Iceberg의 경우 소스의 metadata 파일도 늘어나 plan overhead가 생깁니다. 해결 방법은 `rewriteDataFiles` compaction으로 파일을 병합하거나, write 시 `coalesce()` 또는 `write.target-file-size-bytes` 설정으로 파일 크기를 128~256MB 수준으로 조정합니다.

- 18-7. `watermark`를 Spark Streaming에서 설정하는 이유는? 설정하지 않으면 어떤 문제가 생기나요?

✅ Watermark는 이벤트 시간 기반 집계에서 "얼마나 늦게 도착한 데이터까지 허용할지"를 정의합니다. Watermark를 설정하지 않으면 Spark는 늦게 도착한 데이터를 처리하기 위해 이전 윈도우의 상태를 무기한 메모리에 유지합니다. 결국 상태가 무한히 쌓여 OOM이 발생합니다. `withWatermark("event_time", "10 minutes")`처럼 설정하면 10분보다 늦게 도착한 이벤트는 무시하고, 그 이전 상태를 메모리에서 제거해 안정적으로 운영할 수 있습니다.

## PART 9. 인성 & 조직 핏

### Q19. 셔클 데이터팀의 업무 중 "데이터 사이언티스트의 모델 개발 생산성 향상을 위한 Feature Store 설계"가 있습니다. 직접 모델을 개발해본 경험이 없는 데이터 엔지니어로서, DS팀의 니즈를 어떻게 파악하고 Feature Store를 설계하시겠습니까?

✅ 먼저 DS팀의 실제 워크플로우를 직접 관찰하겠습니다. 어떤 피처를 어떻게 뽑고, 학습과 서빙에서 각각 어떻게 사용하는지 이해하는 것이 설계의 출발점입니다. 이를 토대로 **오프라인 스토어**(학습용 배치 피처 — 기존 Iceberg 활용), **온라인 스토어**(서빙용 저지연 조회 — Redis/DynamoDB), **피처 레지스트리**(메타데이터, 버전, 오너십 관리)의 3개 레이어로 설계하겠습니다. 처음부터 완벽한 시스템보다, DS팀이 가장 자주 재계산하는 피처 3~5개를 Feature Store로 이관하는 quick win으로 시작해 점진적으로 확장하겠습니다.

**꼬리질문**

- 19-1. DS팀과 협업 시 발생할 수 있는 "엔지니어링 우선 vs 빠른 실험 우선" 관점 충돌을 어떻게 해결하시겠나요?

✅ 두 요구를 공간적으로 분리하겠습니다. **샌드박스 환경**에서는 DS팀이 자유롭게 피처를 실험할 수 있도록 제약을 최소화하고, **프로덕션 환경**으로 승격할 때는 데이터 품질 체크, 스키마 등록, SLA 정의를 거치는 프로세스를 만듭니다. "실험은 자유롭게, 프로덕션은 기준을 가지고"라는 원칙으로, 둘 다 만족하는 구조를 설계하겠습니다.

- 19-2. Feature Store 도입 초기에 DS팀이 잘 쓰지 않는다면 어떻게 도입을 촉진하시겠습니까?

✅ 강제가 아니라 페인 포인트 해결로 접근하겠습니다. DS팀이 현재 가장 불편하게 느끼는 것 — 예를 들어 "같은 피처를 팀원마다 다르게 계산하는 문제" 또는 "학습 피처와 서빙 피처가 달라 모델 정확도가 떨어지는 문제" — 을 Feature Store로 해결해 보여주겠습니다. 또한 기존 노트북 코드 대비 Feature Store SDK를 쓰는 것이 더 간단하도록 추상화해, 도구를 쓰는 것 자체가 편리하게 만드는 것이 핵심입니다.

### Q20. 현대자동차라는 대기업 조직과 스타트업(트래블월렛, 줌인터넷)은 의사결정 속도와 프로세스가 다릅니다. 이 차이를 어떻게 적응하시겠습니까?

✅ 대기업의 프로세스를 속도를 느리게 만드는 장애물이 아니라 품질과 신뢰를 보장하는 체계로 인식하겠습니다. 스타트업에서 빠른 결정을 해온 경험이 있지만, 그 결정들을 근거 있게 문서화하고 설명하는 습관도 동시에 길러왔습니다. 대기업 환경에서는 이 문서화 역량이 오히려 빠른 합의를 이끌어내는 도구가 될 수 있다고 봅니다. 초기에는 기존 프로세스를 충분히 이해하고 따르면서 신뢰를 쌓고, 개선 제안은 그 이후에 순서를 밟아 하겠습니다.

**꼬리질문**

- 20-1. 스타트업에서는 빠른 결정이 장점이었지만, 셔클처럼 안전과 연결된 모빌리티 서비스에서는 신중함이 요구됩니다. 이 균형을 어떻게 맞추시겠나요?

✅ 변경의 **blast radius(영향 범위)**를 기준으로 신중함의 수준을 조절하겠습니다. 운영 배차 파이프라인이나 실시간 서빙에 영향을 주는 변경은 충분한 테스트와 단계적 배포(카나리 배포, 병렬 운영)를 거칩니다. 반면 분석용 배치 파이프라인이나 BI 관련 변경은 상대적으로 빠르게 진행할 수 있습니다. 모든 것을 동일한 속도로 처리하는 것이 아니라, 변경이 미치는 위험도에 따라 속도를 다르게 적용하겠습니다.
