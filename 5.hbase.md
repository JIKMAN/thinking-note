# HBase

---

## 아키텍처

```
Client
  ↓
ZooKeeper          → HMaster 위치, Region 메타데이터 캐시
  ↓
HMaster            → Region 할당, 부하 분산, DDL 처리
  ↓
RegionServer × N   → 실제 읽기/쓰기 처리
  ↓
HDFS               → 영구 저장소 (HFile, WAL)
```

### RegionServer 내부 구조

```
RegionServer
  └── Region (테이블의 row key 범위 단위 파티션)
        └── Store (Column Family 단위)
              ├── MemStore  → 쓰기 버퍼 (in-memory)
              └── HFile × N → 영구 저장 파일 (HDFS)
```

**쓰기 흐름**
1. WAL(Write-Ahead Log)에 먼저 기록 → 장애 복구용
2. MemStore에 기록
3. MemStore가 임계치(기본 128MB)를 넘으면 HFile로 flush (HDFS에 저장)
4. HFile이 쌓이면 Compaction으로 병합

**읽기 흐름**
1. BlockCache(RegionServer 메모리) 확인
2. MemStore 확인
3. HFile 확인 (Bloom Filter로 불필요한 HFile 스킵)

## 핵심 개념

### Row Key 설계
HBase의 모든 것이 Row Key에서 시작함. 성능과 데이터 분산이 Row Key에 달려 있음.

- 데이터는 Row Key 기준으로 **사전순 정렬**되어 저장됨
- 범위 스캔(Scan)은 Row Key가 연속된 경우에만 효율적
- 특정 Row Key를 정확히 알면 Get 연산으로 O(1)에 가까운 속도

**Row Key 설계 원칙**
- 조회 패턴에 맞춰 설계. HBase는 Row Key 이외의 컬럼에 인덱스가 없음
- 자주 함께 조회되는 데이터는 Row Key가 연속되도록 설계

### Hotspotting
순차 증가하는 Row Key(timestamp, auto-increment ID 등)는 모든 쓰기가 마지막 Region에 집중됨 → 특정 RegionServer 과부하

**회피 방법**
```
# 1. Salt: Row Key 앞에 해시 prefix 추가
salt = hash(original_key) % N
row_key = f"{salt}_{original_key}"

# 2. Reverse: timestamp를 뒤집음
row_key = str(Long.MAX_VALUE - timestamp) + "_" + user_id

# 3. UUID: 랜덤 분산 (범위 스캔이 필요 없을 때)
```

### Column Family
- HBase 스키마의 기본 단위. 테이블 생성 시 미리 정의해야 함
- 같은 Column Family 내의 컬럼들은 같은 HFile에 저장됨 → 함께 읽히는 컬럼은 같은 CF에
- Column Family 수는 최소화 (2~3개 이내 권장). CF가 많을수록 HFile, MemStore가 늘어남

### Versioning
- HBase는 같은 셀(Row + Column)에 대해 **여러 버전**을 타임스탬프와 함께 저장함
- 기본 버전 수: 1 (최신 1개만 보존)
- `VERSIONS => 3` 설정 시 최근 3개 버전 보존
- 이력 관리, 시계열 데이터에 활용 가능

### WAL (Write-Ahead Log)
- 모든 쓰기는 MemStore에 반영되기 전에 WAL에 먼저 기록
- RegionServer 장애 시 WAL을 재생해서 MemStore에 없던 데이터 복구
- 쓰기 성능이 중요하고 데이터 유실이 허용되는 경우: `setDurability(Durability.SKIP_WAL)` (위험)

### Compaction
HFile이 쌓이면 읽기 성능이 저하됨. Compaction으로 HFile을 병합해서 최적화.

| 종류 | 동작 |
|---|---|
| Minor Compaction | 작은 HFile 몇 개를 병합. 자동으로 자주 실행. 부하 낮음 |
| Major Compaction | 한 Region의 모든 HFile을 하나로 병합 + 삭제 마킹된 데이터 실제 삭제. 부하 큼 |

- Major Compaction은 기본적으로 7일마다 자동 실행. 운영 환경에서는 자동 비활성화하고 새벽에 수동 실행 권장
- Compaction 중에는 CPU, 디스크 I/O, 네트워크 부하 급증

### Region Split
- Region이 일정 크기(기본 10GB)를 넘으면 자동으로 두 개로 분할
- Split 직후 HMaster가 새 Region을 RegionServer에 재할당하는 동안 일시적 지연 발생 (Split Storm)

**Pre-splitting**
- 테이블 생성 시 미리 Region을 나눠두면 Split Storm 방지
- Row Key 분포를 알고 있으면 Pre-splitting이 훨씬 효율적

```java
// 예시: salt 0~9 기반 10개 Region으로 Pre-split
byte[][] splits = new byte[9][];
for (int i = 1; i <= 9; i++) {
    splits[i-1] = Bytes.toBytes(String.valueOf(i));
}
admin.createTable(tableDescriptor, splits);
```

### BlockCache
- RegionServer의 읽기 캐시. 자주 읽히는 HFile 블록을 메모리에 올려둠
- 기본: LRU Cache. 읽기 패턴이 특정 Row에 집중되면 효과가 큼
- 전체 스캔(Scan) 위주라면 캐시 적중률이 낮음 → `setCacheBlocks(false)` 옵션으로 캐시를 우회해서 다른 데이터 캐시 보호

## 실무에서 중요한 개념

### HBase vs RDBMS 비교

| 항목 | HBase | RDBMS |
|---|---|---|
| 스키마 | 유연 (컬럼 동적 추가) | 엄격 |
| 인덱스 | Row Key만 | 다중 인덱스 |
| 조인 | 지원 안 함 | 지원 |
| 트랜잭션 | Row 단위 원자성만 | 완전한 ACID |
| 용도 | 대용량 단순 조회/쓰기 | 복잡한 쿼리, 관계형 데이터 |

### HBase가 적합한 유스케이스
- 단일 Row Key로 조회하는 패턴 (사용자별 이벤트 로그, 센서 데이터 등)
- 수십억 건 이상의 대용량 데이터
- 쓰기가 많고 읽기는 특정 키로 조회하는 패턴
- 컬럼이 동적으로 추가되는 구조

### HBase가 맞지 않는 유스케이스
- 복잡한 조건 필터링 (Row Key 이외의 컬럼으로 검색)
- 집계 쿼리 (SUM, AVG, GROUP BY) → HBase 위에서 Spark로 처리하거나 Hive 연동
- 다중 Row 트랜잭션
- 조인이 필요한 관계형 데이터

### Phoenix (SQL on HBase)
- HBase 위에서 SQL을 쓸 수 있게 해주는 레이어
- Secondary Index 지원 → Row Key 외 컬럼으로 조회 가능
- 단, Secondary Index는 별도 HBase 테이블로 구현되므로 쓰기 비용 증가

## 실제 운영 주의사항

### Row Key 설계는 배포 전에 확정
- Row Key는 나중에 바꾸기 매우 어려움. 바꾸려면 전체 데이터 재적재 필요
- 조회 패턴이 바뀌면 Row Key도 바꿔야 함 → 설계 단계에서 조회 패턴을 명확히 확정

### Major Compaction 자동 실행 비활성화
- 자동 Major Compaction이 운영 시간에 실행되면 I/O 폭증으로 전체 클러스터 성능 저하
- `hbase.hregion.majorcompaction=0` 으로 자동 비활성화
- Cron으로 새벽 트래픽이 낮은 시간에 수동 실행

```bash
echo "major_compact 'table_name'" | hbase shell
```

### GC 모니터링
- RegionServer는 JVM 위에서 동작. Full GC가 발생하면 수 초~수십 초간 응답 없음
- MemStore + BlockCache가 힙의 대부분을 차지함
- G1GC 권장. GC pause time 200ms 초과 시 알림 설정
- RegionServer 힙은 보통 32GB 이하로 설정 (힙이 너무 크면 GC 시간 증가)

### Region 불균형 모니터링
- 특정 RegionServer에 Region이 몰리면 핫스팟 발생
- HMaster UI 또는 `hbase hbck`으로 Region 분포 확인
- `hbase shell`에서 `balance_switch true` + `balancer`로 재분산

### HBCK (HBase Consistency Check)
- Region 메타데이터와 실제 HDFS 파일 간 불일치를 점검하는 도구
- 클러스터 이상 증상 시 첫 번째로 실행할 진단 도구
- HBase 2.x부터는 `hbck2` 사용

```bash
hbase hbck -details
```

### ZooKeeper 세션 타임아웃
- RegionServer가 ZooKeeper에 일정 시간 응답 못 하면 세션 만료 → HMaster가 해당 RegionServer를 Dead로 판단 → Region 재할당
- GC pause가 길면 ZooKeeper 세션 타임아웃으로 이어질 수 있음
- `zookeeper.session.timeout` 값을 GC pause 예상 시간보다 충분히 크게 설정

### 쓰기 버퍼 활용
- 대량 쓰기 시 매 건마다 RPC를 보내면 오버헤드가 큼
- Client 버퍼링을 활성화하거나 `BufferedMutator`를 사용해서 배치로 묶어서 전송
- `hbase.client.write.buffer` 설정으로 버퍼 크기 조정

### 스냅샷 기반 백업
- HBase 스냅샷은 HDFS 파일을 복사하지 않고 메타데이터만 찍음 → 빠르고 스토리지 효율적
- 스냅샷 → Export Snapshot으로 다른 클러스터나 S3로 복사 가능
- 주기적 스냅샷을 운영 루틴에 포함시키는 것이 안전

```bash
echo "snapshot 'table_name', 'snapshot_name'" | hbase shell
hbase org.apache.hadoop.hbase.snapshot.ExportSnapshot \
  -snapshot snapshot_name -copy-to hdfs://backup-cluster/hbase
```
